# 技巧一：分层组装 & 缓存优化

> 核心思想：不是写一段大 prompt，而是把 prompt 拆成可独立缓存的层

---

## 问题

LLM API 支持 prompt caching（前缀缓存），即：如果连续两次请求的 prompt 开头相同，API 可以跳过重复部分的计算。但如果 prompt 每次都变（比如包含当前时间、文件路径），缓存就失效了。

## Claude Code 的解法

### 1. 静态/动态分界线

在 `src/constants/prompts.ts` 中定义了一个关键常量：

```
SYSTEM_PROMPT_DYNAMIC_BOUNDARY
```

整个系统 prompt 被这条线分成两部分：

```
┌─────────── 静态层（Global Scope，可缓存）───────────┐
│  身份定义                                          │
│  安全指令                                          │
│  行为准则                                          │
│  编码风格                                          │
│  工具使用指南                                      │
├─────── SYSTEM_PROMPT_DYNAMIC_BOUNDARY ─────────────┤
│  当前会话可用的工具列表      ← 每次请求可能不同      │
│  环境信息（CWD, git, OS）   ← 每次会话不同          │
│  模型信息                   ← 可能切换模型          │
│  会话级 guidance             ← 因工具可用性变化      │
└─────────── 动态层（Session-Specific）───────────────┘
```

**效果**: 静态层在同一次会话的多次 API 调用之间完全复用，极大提升缓存命中率。

### 2. 分段缓存（System Prompt Sections）

`src/constants/systemPromptSections.ts` 实现了更细粒度的缓存：

```typescript
// 可缓存的分段 —— memoized，直到 /clear 或 /compact 才刷新
systemPromptSection(() => getSimpleIntroSection())
systemPromptSection(() => getSimpleSystemSection())
systemPromptSection(() => getSimpleDoingTasksSection())

// 不可缓存的分段 —— 每次都重新计算
DANGEROUS_uncachedSystemPromptSection(() => computeSimpleEnvInfo())
```

每个 section 是一个**无参函数**，返回 prompt 字符串。`systemPromptSection()` 对其做 memoize（记忆化）——同一会话内只计算一次，后续直接返回缓存值。

`DANGEROUS_uncachedSystemPromptSection()` 前缀故意用了 "DANGEROUS" 命名，提醒开发者这个 section 会**打破缓存**，每次调用都重新计算。

### 3. 优先级覆盖机制

`src/utils/systemPrompt.ts` 定义了 prompt 覆盖的优先级链：

```
override > coordinator > agent > custom > default + appendSystemPrompt
```

- **default**: 标准系统 prompt（`getSystemPrompt()` 生成）
- **custom**: 用户通过配置自定义的 prompt
- **agent**: 子 Agent 使用的专用 prompt
- **coordinator**: 多 Agent 协调器使用的 prompt
- **override**: 完全替换系统 prompt（最高优先级）

`appendSystemPrompt` 可以在任何层级**追加**内容，而不是替换。

---

## 提示词工程技巧总结

| 技巧 | 具体做法 | 效果 |
|------|---------|------|
| 静态/动态分界 | 用 `DYNAMIC_BOUNDARY` 标记分割 | 静态部分跨请求缓存 |
| 分段 memoize | 每个 section 是无参函数，被 `systemPromptSection()` 包装 | 细粒度缓存控制 |
| 危险标记命名 | `DANGEROUS_uncached...` 前缀 | 防止开发者误用打破缓存 |
| 无参函数模式 | section 函数不接收参数 | 保证幂等性，安全缓存 |

## 给抖音文案的启发

> "Claude 不是把所有指令塞进一段话。它像搭积木一样，把不变的指令放在前面缓存，把会变的信息放在后面。这样每次对话只需要重新计算一小部分——省下的算力，就是省下的钱和时间。"
