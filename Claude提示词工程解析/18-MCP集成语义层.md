# 技巧十八：MCP 工具注解——不用猜的权限

> 核心思想：让工具通过协议层声明自己的危险程度，而不是让模型从描述中"猜"——元数据 > 自然语言

---

## 问题本质

当 Claude Code 接入外部 MCP 工具时，一个关键问题出现了：**这个工具是只读的还是破坏性的？**

靠模型从工具描述中判断？

```
工具描述："删除指定资源"
→ 模型理解：可能危险
→ 但"删除"一个临时缓存文件 vs "删除"生产数据库，模型怎么区分？
```

靠用户手动配置权限？

```
每个 MCP 工具都需要在 settings.json 中写规则
→ 维护成本高，容易遗漏
→ 新工具接入要等用户配置
```

正确的做法：**让工具自己声明语义**。不是用自然语言描述，而是用结构化的元数据标注。

## 工具注解：四个 Hint 如何让权限系统自动判断

**源码**：MCP schema ~line 5720

MCP 协议允许工具声明四个注解（annotations）：

| 注解 | 类型 | 含义 | 权限影响 |
|------|------|------|---------|
| `readOnlyHint` | boolean | 工具是否只读 | `true` → 自动降低权限等级 |
| `destructiveHint` | boolean | 工具是否可能造成不可逆破坏 | `true` → 自动提升确认要求 |
| `idempotentHint` | boolean | 重复调用是否产生相同结果 | `true` → 可安全重试 |
| `openWorldHint` | boolean | 工具是否与外部世界交互 | `true` → 需要额外审查 |

### 权限自动升级流程

```
工具声明 annotations
    │
    ├── readOnlyHint: true
    │   → 权限系统：自动放行（只读操作不修改任何东西）
    │
    ├── destructiveHint: true
    │   → 权限系统：要求确认（或直接触发 auto 分类器）
    │
    ├── idempotentHint: true
    │   → 权限系统：失败可安全重试
    │
    └── openWorldHint: true
        → 权限系统：有外部影响，需要更严格的审查
```

**对比：没有注解的时代**

```
❌ 模型读工具描述："Deletes a record from the database"
   → 模型判断："看起来危险"
   → 但不同模型判断不一致
   → 提示词需要反复强调"要注意删除操作"

✅ 工具声明 { destructiveHint: true }
   → 权限系统直接升级确认级别
   → 不依赖模型判断
   → 提示词不需要提这件事
```

### 注解的组合语义

| readOnlyHint | destructiveHint | 实际含义 | 权限级别 |
|-------------|----------------|---------|---------|
| true | false | 纯只读（如 `git status`） | 最低 |
| true | true | 矛盾声明，按 destructive 处理 | 高 |
| false | false | 有修改但非破坏性（如 `git add`） | 中 |
| false | true | 破坏性操作（如 `git reset --hard`） | 最高 |

## claudeai-proxy：云端工具零本地进程

MCP 工具有一个特殊类型：`claudeai-proxy`。

```
普通 MCP 工具：
Claude Code → stdio/SSE → MCP Server（本地进程）

claudeai-proxy：
Claude Code → 云端 → MCP Server（服务器端）
```

**特性**：
- **无本地进程**：工具在云端运行，不需要用户本地启动任何服务
- **零配置**：不需要安装、配置 MCP 服务器
- **动态发现**：工具列表从云端获取，随时更新

**实际意义**：这意味着 Claude Code 可以接入云端的工具生态——就像浏览器插件一样，不需要本地部署。

## Elicitation：MCP 服务器的"提问权"

传统模式下，只有用户能提问，AI 能回答，工具只是被动执行。

MCP 的 `Elicitation` 机制打破了这个单向关系：

```
MCP Server（工具端）
    │
    ├── 需要额外信息才能执行操作
    │
    ▼
Elicitation 请求
    │
    ├── 方式1：表单（结构化字段）
    │   { "type": "form", "fields": [
    │     { "name": "target", "label": "部署目标", "options": ["staging", "prod"] }
    │   ]}
    │
    ├── 方式2：URL 链接
    │   { "type": "url", "url": "https://auth.example.com/approve" }
    │
    ▼
用户 / Hooks 自动响应
    │
    ▼
MCP Server 收到响应，继续执行
```

**Hooks 自动响应**：结合技巧十六的 Hooks 系统，`Elicitation` 事件可以被钩子自动处理：

```json
{
  "hooks": {
    "Elicitation": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "echo '{\"response\": \"approved\", \"reason\": \"白名单操作\"}'"
          }
        ]
      }
    ]
  }
}
```

**典型场景**：
- OAuth 授权流程：工具需要用户点击链接完成授权
- 部署确认：工具在执行部署前弹出表单让用户选择目标环境
- 费用确认：付费操作前让用户确认金额

## Sampling：MCP 服务器的"借脑权"

Sampling 是另一个打破单向关系的机制——MCP 服务器可以请求 Claude 生成文本：

```
MCP Server
    │
    ├── 需要 AI 能力但不想自己调用 API
    │
    ▼
Sampling 请求
{
  "messages": [{"role": "user", "content": "总结这段代码的功能"}],
  "modelPreferences": {"speed": "fast"},
  "maxTokens": 500
}
    │
    ▼
Claude Code 代理执行（使用当前会话的模型）
    │
    ▼
返回生成结果给 MCP Server
```

**实际场景**：
- 代码审查工具：拿到代码后，请求 AI 生成审查意见
- 文档生成工具：拿到 API schema 后，请求 AI 生成文档
- 测试生成工具：拿到函数签名后，请求 AI 生成测试用例

**安全考量**：Sampling 请求经过 Claude Code 代理，受权限系统管控——不是 MCP 服务器想调就调。

## 动态注入：MCP 工具如何进入系统提示词

MCP 工具不是静态配置的——它们在运行时动态注入到系统提示词中。

**源码**：prompt injection ~line 997/1260

```
系统提示词结构：
┌──────────────────────────────────┐
│  静态层（缓存）                    │
│  - 身份定义                        │
│  - 安全规则                        │
│  - 工具使用指南                    │
├──────────────────────────────────┤
│  动态层（不缓存）                  │
│  - 环境信息                        │
│  - CLAUDE.md 内容                 │
│  - **Configured MCP servers:**   │ ← MCP 工具在这里注入
│    - server1: tool_a, tool_b     │
│    - server2: tool_c             │
├──────────────────────────────────┤
│  对话历史                         │
└──────────────────────────────────┘
```

### 注入格式

```
**Configured MCP servers:**

- **my-server** (connected)
  - tool_a: 描述...
  - tool_b: 描述...
```

**Token 成本**：每个 MCP 工具的名称 + 描述会消耗 prompt tokens。如果有 20 个 MCP 工具，可能增加 2-5K tokens。

**优化手段**：
- 工具描述保持简洁
- 不需要的工具及时断开连接
- 利用 prompt caching（MCP 工具列表在静态层之后，可以被缓存）

## 语义元数据 vs 自然语言：为什么协议层 > 提示词层

| 方式 | 可靠性 | Token 成本 | 可维护性 | 可扩展性 |
|------|--------|-----------|---------|---------|
| 自然语言描述 | 低（模型可能误解） | 高（每轮消耗） | 差（描述可能过时） | 差（N 个工具 N 份描述） |
| 提示词规则 | 中（模型可能忽略） | 高（每轮消耗） | 中（需要手动维护） | 中（规则可能冲突） |
| 工具注解 | 高（结构化，确定性） | 低（一次声明） | 好（工具自己维护） | 好（新增工具自带注解） |
| Hooks 拦截 | 最高（程序化执行） | 零（触发时才执行） | 最好（代码即规则） | 好（规则独立） |

**演进方向**：

```
第一代：靠提示词告诉模型"注意安全"
     → 模型可能忘记，每轮都消耗 token

第二代：靠规则白名单管理权限
     → 需要人工维护，新工具要手动配置

第三代：靠工具注解自动升级权限
     → 工具自带语义声明，零配置

第四代：Hooks + 注解 + ML 分类器联动
     → 协议层、程序层、概率层三位一体
```

Claude Code 正处于第三代到第四代的过渡阶段。

---

## 关键洞察

> **最好的安全约束不是让模型"理解"危险，而是让协议"声明"危险——元数据比自然语言可靠，代码比提示词可靠。**

当你设计 AI 工具集成时，问自己三个问题：
1. 这个信息能不能用结构化元数据表达，而不是自然语言？（能 → 用注解）
2. 这个约束能不能用代码执行，而不是靠模型遵循？（能 → 用 Hooks）
3. 这个判断能不能交给独立系统，而不是让主模型自己决定？（能 → 用分类器）

协议层 > 提示词层，代码 > 描述，独立 > 自律。
