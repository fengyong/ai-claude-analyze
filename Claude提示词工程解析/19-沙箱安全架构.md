# 技巧十九：沙箱——OS 级替代提示词约束

> 核心思想：别靠 AI 自律，用操作系统的笼子关住它

---

## 问题本质

提示词安全有三层信任等级：

1. **"请不要做 X"** —— 靠模型自律，一念之差就破防
2. **"用户是否批准 X"** —— 靠用户判断，但弹窗疲劳会让用户无脑点确认
3. **"OS 不允许做 X"** —— 物理铁壁，不管模型想不想做，都做不了

Claude Code 的沙箱是第三种——**把安全从"提示词约束"升级为"操作系统约束"**。

类比：提示词约束就像在门口贴一张"请勿入内"的纸条，沙箱是直接把门焊死。

---

## 三级纵深防御架构

```
┌─────────────────────────────────────────────────────┐
│                第一层：提示词层（最弱）                 │
│  "Do not modify .git/hooks or .git/config"          │
│  特点：模型依赖、可被注入覆盖、无强制力               │
├─────────────────────────────────────────────────────┤
│                第二层：权限层（中等）                   │
│  工具调用 → 弹窗请求用户确认                          │
│  特点：用户依赖、弹窗疲劳、社会工程攻击               │
├─────────────────────────────────────────────────────┤
│                第三层：沙箱层（最强）                   │
│  OS 内核强制执行 denyWrite / denyRead                │
│  特点：物理隔离、无弹窗、不可绕过                     │
└─────────────────────────────────────────────────────┘
```

**核心转变**：当沙箱启用后，第一层和第二层的压力大幅减轻——很多"请不要做"的提示词可以直接删掉，很多弹窗确认可以直接跳过。**沙箱释放了上下文窗口的 token 预算。**

## macOS vs Linux 沙箱实现差异

### macOS：Apple Sandbox Profile

macOS 使用内核级沙箱（`sandbox-exec`），基于 Seatbelt 策略语言：

```
; deny-default 策略示例
(version 1)
(deny default)
(allow process-exec)
(allow file-read* (subpath "/Users"))
(allow file-write* (subpath "/tmp/claude-code"))
(deny file-write* (subpath "/Users/project/.git"))
(allow network-outbound (to domain "api.anthropic.com"))
(deny network-outbound)
```

**源码**：`cli.js ~line 12032` — 沙箱 schema 定义（`$x7`）

沙箱配置使用 JSON schema 定义，核心字段：

```json
{
  "sandbox": {
    "enabled": true,
    "mode": "deny-default",
    "permissions": {
      "denyRead": ["~/.ssh/**", "**/credentials.json"],
      "allowRead": [".", "/tmp"],
      "denyWrite": ["**/.git/hooks/**", "**/.git/config", "node_modules/.cache"],
      "allowWrite": ["src/**", ".claude/**"]
    },
    "network": {
      "allowDomains": ["api.anthropic.com", "registry.npmjs.org"],
      "denyDomains": ["*"]
    }
  }
}
```

### Linux：Bubblewrap + Seccomp

Linux 使用 `bubblewrap`（容器化隔离）+ `seccomp`（系统调用过滤）的组合：

```
bwrap --ro-bind /usr /usr \
      --bind /tmp/claude-work /work \
      --unshare-net \
      --seccomp 3 3<seccomp.bpf \
      -- /bin/bash
```

| 维度 | macOS | Linux |
|------|-------|-------|
| 隔离机制 | Apple Sandbox Profile | Bubblewrap + Seccomp |
| 网络控制 | 策略级域名过滤 | 网络命名空间隔离 |
| 文件系统 | 路径规则（allow/deny） | 挂载点绑定 |
| 内核接口 | `sandbox-exec` | `bwrap` + `prctl` |
| 性能开销 | 极低（内核原生） | 极低（namespace） |

**源码**：`cli.js ~line 29878` — 沙箱启动流程

## autoAllowBashIfSandboxed：沙箱如何反向简化权限

这是最精妙的设计——**沙箱不仅约束了危险操作，还消除了安全弹窗的必要性**。

```typescript
// 核心逻辑
autoAllowBashIfSandboxed: true  // 沙箱启用时，Bash 自动批准
```

推理链条：

```
前提 1：沙箱限制了 Bash 的破坏能力（不能读 .ssh，不能写 .git）
前提 2：用户已经信任沙箱策略（启动时检查过）
结论：沙箱内的 Bash 不再需要逐次确认
```

**效果对比**：

| 场景 | 无沙箱 | 有沙箱 |
|------|--------|--------|
| `npm install` | 弹窗确认 | 自动执行 |
| `curl api.example.com` | 弹窗确认 | 自动执行（域名在白名单） |
| `rm -rf .git` | 弹窗确认 | 沙箱直接拒绝，无需弹窗 |
| `cat ~/.ssh/id_rsa` | 弹窗确认 | 沙箱直接拒绝，无需弹窗 |

**类比**：游乐场的儿童区——如果没有围栏，你得寸步不离地盯着孩子（弹窗确认）。有了围栏，孩子可以在里面自由奔跑（自动批准），因为围栏已经挡住了所有危险。

## Go 工具的 TLS 问题：enableWeakerNetworkIsolation 的务实妥协

### 问题

Go 语言编写的 CLI 工具（`gh`、`gcloud`、`terraform`、`kubectl`）在 macOS 上有一个特殊的 TLS 行为——它们使用 `com.apple.trustd.agent` 服务来验证证书，而不是直接读取系统证书存储。

沙箱的网络隔离会阻断这个服务调用，导致：

```
$ gh pr list
Error: x509: certificate signed by unknown authority
```

### 解决方案

```json
{
  "sandbox": {
    "enableWeakerNetworkIsolation": true
  }
}
```

这个标志允许 `com.apple.trustd.agent` 通信，但保持其他网络限制。

| 维度 | 严格网络隔离 | 弱化网络隔离 |
|------|------------|------------|
| 外部 HTTP | 拒绝 | 拒绝 |
| DNS 查询 | 拒绝 | 拒绝 |
| trustd 通信 | 拒绝 | **允许** |
| 适用场景 | Node.js / Python 工具 | Go CLI 工具 |

**这是工程上的务实选择**：不是完美的安全，但比"不用沙箱"好得多。安全不是一个二元选择，而是一个光谱。

## 网络控制：域名白名单与 MITM 代理

沙箱的网络层支持精细的域名控制：

```json
{
  "network": {
    "allowDomains": [
      "api.anthropic.com",     // Claude API
      "registry.npmjs.org",    // 包管理
      "github.com",            // 代码托管
      "api.github.com"         // GitHub API
    ],
    "mitmProxy": {
      "enabled": true,
      "certPath": "/etc/corp-ca.pem"
    }
  }
}
```

MITM 代理支持允许企业在不解除网络隔离的情况下进行流量审计——安全团队可以看到 AI 发了什么请求，但 AI 本身不能随意访问互联网。

---

## 关键洞察

沙箱不是"更严格的提示词"，而是"不需要提示词"——当操作系统保证 AI 不能写 .git/hooks，你就不必浪费 token 告诉它不要写。
