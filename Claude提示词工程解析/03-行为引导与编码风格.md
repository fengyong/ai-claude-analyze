# 技巧三：行为引导 & 编码风格

> 核心思想：不只是说"写好代码"，而是精确到每个行为细节

---

## 七大行为准则 Section

`src/constants/prompts.ts` 通过 7 个独立函数构建行为准则，每个函数返回一个 prompt 片段：

### 1. `getSimpleDoingTasksSection()` —— 执行任务时的行为

**反过度工程化规则**：
- 不添加 "error handling, fallbacks, or validation for scenarios that can't happen"
- 不添加 "feature flags or backwards-compatibility shims when you can just change the code"
- "Three similar lines of code is better than a premature abstraction"
- "Don't create helpers, utilities, or abstractions for one-time operations"

**安全规则**：
- "Be careful not to introduce security vulnerabilities such as command injection, XSS, SQL injection"
- "If you notice that you wrote insecure code, immediately fix it"

**边界**：
- 只在系统边界（用户输入、外部 API）做验证
- 不为不可能发生的场景添加错误处理

### 2. `getActionsSection()` —— 操作风险评估

**可逆性框架**：
- "Carefully consider the reversibility and blast radius of actions"
- 列出高风险操作的具体示例：删除文件/分支、drop 表、kill 进程、`rm -rf`
- "measure twice, cut once" —— 确认成本低，误操作成本高

**确认原则**：
- 默认需要确认（不因用户之前批准过一次就认为永久授权）
- 只在用户明确授权范围外的操作才需要确认

### 3. `getUsingYourToolsSection()` —— 工具使用偏好

**工具优先级**：
```
专用工具 > Bash
Read > cat/head/tail/sed
Write > echo/heredoc
Glob > find/ls
Grep > grep/rg
```

**并行执行**：
- 无依赖的工具调用必须并行
- "Maximize use of parallel tool calls where possible"

**子代理使用**：
- 匹配 Agent 描述时主动使用，不需要用户请求
- 避免重复子代理已经在做的事

### 4. `getSimpleToneAndStyleSection()` —— 语气风格

**核心规则**：
- **不要用 emoji**（除非用户明确要求）
- 回复简短直接
- 引用代码时用 `file_path:line_number` 格式
- 一句能说清的，不用三句

**输出效率**：
- "Go straight to the point"
- 不重述用户说了什么
- 只在需要用户决策、高里程碑、遇到错误时才多说

### 5. `getOutputEfficiencySection()` —— 输出效率（Ant vs External 差异）

**外部用户版本**：
- 简洁导向：不要废话

**Anthropic 内部用户版本**：
- 包含写作指导（prose-writing guidance）
- 更详细的风格建议

### 6. `getSessionSpecificGuidanceSection()` —— 会话级动态指导

根据当前可用的工具动态调整指导内容：

| 可用工具 | 触发的指导 |
|---------|-----------|
| `AskUserQuestion` | 何时提问、何时不问 |
| `Agent` | 子代理使用策略 |
| `Skill` | 技能调用规则 |
| `Verification` | 验证流程 |

### 7. `computeSimpleEnvInfo()` —— 环境信息

实时计算，包含：
- 当前工作目录
- Git 状态（是否是 git 仓库）
- 平台（darwin/linux/win32）
- Shell（bash/zsh/fish）
- 模型名称和知识截止日期

## 编码风格规则

在 prompt 中硬编码的风格规则：

```
- Linter: Biome (not ESLint)
- Indentation: Tabs, 100-char line width
- Quotes: Single quotes, no semicolons
- Imports: ESM with explicit .js extensions
- Files/dirs: kebab-case
- Classes/components: PascalCase
- Functions: camelCase
- Constants: UPPER_SNAKE_CASE
```

这些规则在 prompt 中用"不要做 X"的否定形式表达，比"应该做 Y"的肯定形式更有效——因为模型需要知道**边界在哪里**。

## 提示词工程技巧总结

| 技巧 | 具体做法 | 效果 |
|------|---------|------|
| 否定优先 | "Don't do X" > "Do Y" | 明确边界比模糊建议更有效 |
| 具体化 | 列出 `rm -rf` 等具体例子 | 比"危险操作"四个字更有指导性 |
| 优先级明确 | 专用工具 > Bash > 什么都不做 | 消除选择歧义 |
| 双版本 | Ant 用户 vs 外部用户不同内容 | 精准匹配用户群体 |
| 动态注入 | 根据可用工具决定加哪些指导 | 不浪费 token 给不可用的工具写指导 |
