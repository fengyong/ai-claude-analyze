# 技巧七：上下文压缩 & 摘要

> 核心思想：对话太长时，用 AI 压缩 AI 的记忆

---

## 问题

LLM 的上下文窗口有限（Claude Opus 4.6 有 200K tokens）。一场长对话可能消耗掉大量 tokens，最终要么超限，要么变得极慢。

## Claude Code 的解法：Compact（压缩）

`src/services/compact/prompt.ts` 实现了上下文压缩系统。

### 压缩前的准备

在压缩开始前，插入一条指令防止浪费：

```
NO_TOOLS_PREAMBLE = "Respond with TEXT ONLY"
```

**Why**: 如果压缩时模型还尝试调用工具，会浪费一轮 API 调用。这条指令直接禁止工具调用。

### 九段式摘要结构

压缩后的摘要必须包含以下 9 个部分：

```
1. Primary Request and Intent
   → 用户最初想要什么，当前想要什么

2. Key Technical Concepts
   → 涉及的技术概念和术语

3. Files and Code Sections
   → 修改过的文件及其作用（含具体行号和代码片段）

4. Errors and fixes
   → 遇到的错误及修复方法

5. Problem Solving
   → 已解决的问题和未解决的问题

6. All user messages
   → 用户所有消息的摘要（按时间顺序）

7. Pending Tasks
   → 还没完成的任务

8. Current Work
   → 压缩前正在进行的工作

9. Next Step
   → 下一步该做什么
```

### `<analysis>` 草稿标签

模型先在 `<analysis>` 标签内打草稿，然后输出最终摘要：

```
<analysis>
  [模型在这里写草稿，整理思路]
</analysis>

1. Primary Request and Intent: ...
2. Key Technical Concepts: ...
...
```

**Why**: `<analysis>` 标签的内容在注入上下文前会被**剥离**（strip），只保留最终摘要。这让模型有"打草稿"的空间，而不污染最终上下文。

### 部分压缩策略

支持两种部分压缩模式：

**`from` 模式**：压缩"从某条消息开始"之后的内容，保留更早的消息
```
[保留的消息...] → [压缩摘要] → [最近的消息...]
```

**`up_to` 模式**：压缩"到某条消息为止"的前缀，保留最近的消息
```
[压缩摘要] → [最近的消息...]
```

---

## 提示词工程技巧总结

| 技巧 | 具体做法 | 效果 |
|------|---------|------|
| No-tools 前言 | "Respond with TEXT ONLY" | 防止压缩时浪费工具调用 |
| 结构化摘要 | 9 个固定 section | 保证摘要完整性，不遗漏关键信息 |
| 草稿标签 | `<analysis>` 标签，输出前剥离 | 给模型思考空间，不污染最终输出 |
| 部分压缩 | `from` / `up_to` 两种模式 | 灵活选择压缩范围 |
| 保留最近 | 总是保留最近的消息不压缩 | 确保当前上下文不丢失 |
