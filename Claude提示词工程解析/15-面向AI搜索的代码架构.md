# 面向 AI 搜索的代码架构：设计模式与搜索效率

> 核心发现：AI 编程助手的搜索限制（250行/100文件/2000行）不是缺陷，是工程约束——它们实际上鼓励更好的代码结构

---

## 问题

如何用更好的设计模式来应对 Claude 的搜索机制？

---

## 搜索工具的硬性限制回顾

| 工具 | 限制 | 影响 |
|------|------|------|
| GrepTool | 250 行匹配，20K 字符 | 关键词泛滥时结果被截断 |
| GlobTool | 100 个文件，100K 字符 | 通用文件名导致发现困难 |
| ReadTool | 2000 行，25K tokens | 大文件需要多次读取 |
| 聚合预算 | 200K 字符/消息 | 并行调用受限 |

---

## 设计模式与搜索效率的对应关系

### 1. 高内聚 + 单一职责 → 搜索精度倍增

**正面模式**：Claude Code 的工具目录结构

```
src/tools/BashTool/
  ├── BashTool.ts      # 工具定义 + 执行逻辑
  ├── UI.tsx           # 终端渲染
  ├── prompt.ts        # 提示词
  ├── permissions.ts   # 权限模型
```

Glob `src/tools/BashTool/**` → 精确命中该工具的所有文件

**反面案例**：5000 行的 `utils.ts`

```
- Read 至少 3 次（每次 2000 行上限）
- Grep 行号跨度大（行 47 和行 4521 的关系不明）
- 整个文件读取消耗 ~100K 字符 → 吃掉一半消息预算
```

**实践数据**：Claude Code 最大的工具（BashTool）18 个文件共 12,429 行，分散在多个文件中。AgentTool 20 个文件共 6,802 行。

---

### 2. 低耦合 → 减少搜索链深度

搜索链 = AI 需要通过多次搜索才能理解代码关系

```
高耦合（5+ 次搜索）：
A.ts → imports B.ts → calls C.ts → uses D.ts → modifies E.ts

低耦合（2 次搜索）：
A.ts → imports B.ts（B 通过清晰接口隔离，无需追踪到 C/D/E）
```

Claude Code 的实现：`buildTool()` 工厂接口
- AI 读 `Tool.ts` 的类型定义 → 直接看具体工具目录
- 不需要追踪到其他工具的实现

---

### 3. 描述性命名 → 决定搜索成败

Claude Code 的命名规范：

| 类别 | 规则 | 示例 |
|------|------|------|
| 文件/目录 | kebab-case | `commit-push-pr.ts` |
| 类/组件 | PascalCase | `BashTool`, `QueryEngine` |
| 函数 | camelCase | `getCommands()`, `buildTool()` |
| 常量 | UPPER_SNAKE_CASE | `ALLOWED_TOOLS` |

**搜索效果**：

| 搜索词 | 效果 |
|--------|------|
| `buildTool\(` | 精确找到所有工具定义 |
| `satisfies Command` | 精确找到所有命令定义 |
| `process()` | 泛滥——多个模块都有同名函数 |
| `handle()` | 泛滥——通用名称，250 行上限迅速触达 |

**双重职责**：函数名同时服务于人类理解 + AI 搜索索引。

---

### 4. 索引导出模式 → 降低首次搜索成本

Claude Code 广泛使用 `index.ts` 再导出：

```
src/tools/index.ts → 导出所有工具的公共 API
src/commands/index.ts → 导出所有命令
src/services/index.ts → 导出所有服务
```

**搜索流程**：

```
第一步：读 index.ts（通常 < 200 行）
       → 快速理解模块提供的所有接口

第二步：按需深入具体实现文件

对比：如果没有 index.ts
       → 从 50 个文件中逐个判断哪些是公共 API
       → 多消耗 10-20 次工具调用
```

**index.ts 就是代码地图**——小入口文件充当搜索-阅读范式的天然入口点。

---

### 5. 文件规模与上下文预算的数学关系

| 文件行数 | 估计大小 | Read 次数 | 消耗预算（约） |
|----------|----------|-----------|----------------|
| 200 行 | ~8KB | 1 次 | ~2K 字符 |
| 500 行 | ~20KB | 1 次 | ~5K 字符 |
| 1000 行 | ~40KB | 1 次 | ~10K 字符 |
| 2000 行 | ~80KB | 1 次（上限） | ~20K 字符 |
| 5000 行 | ~200KB | 3+ 次 | ~50K 字符 |
| 10000 行 | ~400KB | 5+ 次 | ~100K 字符（半条消息预算） |

**黄金区间：200-800 行**
- 单次 Read 即可完整获取
- Grep 上下文在同一个文件内，行号跨度合理
- 多个并行 Read 不会轻易突破 200K 聚合预算

**红线：超过 2000 行的文件会成为 AI 助手的瓶颈。**

---

### 6. 目录结构即代码地图

推荐模式（Claude Code 自身的做法）：

```
src/feature/
  ├── index.ts           # 公共 API 导出（代码地图）
  ├── FeatureName.ts     # 主逻辑
  ├── UI.tsx             # 渲染层
  ├── prompt.ts          # 提示词
  ├── permissions.ts     # 权限
  └── types.ts           # 类型定义
```

Glob `src/feature/**` → 一步返回功能的所有相关文件

---

### 7. CLAUDE.md 的正确使用

CLAUDE.md 是 AI 助手的引导文件：

| 约束 | 值 |
|------|---|
| 单文件最大 | 40,000 字符 |
| @include 递归 | 5 层 |
| MEMORY.md 入口 | 200 行 / 25KB |
| 单个附件 | 200 行 / 4KB |
| 每轮附件上限 | 5 个 |

**最佳实践**：CLAUDE.md 应充当代码地图——列出关键文件和导航路径，而不是包含大量代码。Claude Code 自身的 CLAUDE.md 正是这样做的：

```
## Key Files
| File | Role |
|------|------|
| src/QueryEngine.ts | LLM engine — streaming, tool-call loops |
| src/Tool.ts | Tool types/interfaces, buildTool() factory |
```

## 设计模式总结

| 模式 | 对搜索的影响 | 具体做法 |
|------|-------------|---------|
| 高内聚 | Glob 精确命中目录 | 每个功能一个目录 |
| 低耦合 | 减少搜索链深度 | 清晰接口隔离，避免长链调用 |
| 单一职责 | 文件保持可读规模 | 200-800 行/文件 |
| 描述性命名 | Grep 精确匹配 | 避免 `process()`、`handle()` |
| 索引导出 | 提供代码地图 | 每个目录 `index.ts` |
| 文件拆分 | 利用并行调用 | 大文件拆成多个职责明确的小文件 |

---

## 核心洞察

> **面向 AI 搜索的代码架构 = 面向人类可读性的代码架构。**
>
> 高内聚、低耦合、单一职责、描述性命名——这些不是新概念，是软件工程几十年的积累。AI 的搜索限制只是让违反这些原则的代价变得更明显了。
>
> 能高效服务于人类搜索的代码组织，同样高效服务于 AI 搜索。两者的目 标是一致的。

## 给抖音文案的启发

> "你以为好代码是写给人看的？现在也是写给 AI 看的。Claude 搜代码有三个硬上限：一次最多搜 250 行，一次最多读 2000 行，一次最多看 100 个文件。你的代码越乱，AI 越瞎。但如果你把文件控制在 500 行以内，函数名取得具体一点，每个目录放一个 index.ts——AI 就像开了上帝视角。好代码从来不只是一种美德，现在它是一种生产力。"
