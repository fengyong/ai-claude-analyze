# 技巧二十六：何时打断用户——主动模式的提示词挑战

> 核心思想：让 AI 从"等你问"变成"主动说"——但说多了是骚扰，说少了是失职

---

## 问题本质

传统 AI 助手是**被动响应**的：用户提问，AI 回答。不问就不说话。这在大多数场景下没问题，但有一种场景天然不适合被动——**后台任务**。

想象一个场景：你让 Claude Code 在后台跑一个长达 30 分钟的构建。传统模式下，构建失败了，Claude 不会告诉你——因为你没问。你回来一看，30 分钟白费了。

**主动模式**（Proactive）解决的就是这个问题：让 AI 在后台自主运行，**主动发起对话**，在需要时打断用户。

但这引出了一个经典的提示词工程难题：

> **有用 vs 不烦人——什么时候该打断用户？**

打得太频繁，用户关掉通知；打得太少，错过关键问题。这个平衡不是写一行"适当提醒"就能解决的。

## 主动模式架构

### 门控机制

主动模式不是一个简单的开关，而是被**多层门控**保护：

```
GrowthBook Flag（实验开关）
    ↓
ZDR Check（零数据保留检查）
    ↓
Managed Settings（企业管理配置）
    ↓
isProactiveActive（运行时状态）
    ↓
用户手动启用 / autoEnable 自动激活
```

**源码**：cli.js ~line 3628, ~line 12032

这不是"一个 if 判断"——而是五层条件嵌套。每一层都有不同的理由：

| 门控层 | 作用 | 为什么需要 |
|--------|------|-----------|
| GrowthBook Flag | 灰度发布 | 不是所有用户都该看到这个功能 |
| ZDR Check | 合规检查 | 零数据保留客户不允许后台代理 |
| Managed Settings | 企业管控 | 公司 IT 可以统一关闭 |
| isProactiveActive | 运行时状态 | 功能开了但用户没激活 |
| autoEnable | 自动激活 | 启动时自动进入主动模式（如果用户有权） |

### Tick 定时器

主动模式的核心是一个**定时检查循环**：

```typescript
// 主动模式的"心跳"
tick() {
  // 检查是否需要打断用户
  // 策略决定：现在是不是该说话的时候
}

minTickIntervalMinutes  // 最小检查间隔，控制频率
```

Tick 定时器不等于"每隔 N 分钟说一次话"。它是**检查频率**，不是**发言频率**。每次 tick，代理评估：当前是否有值得打断用户的事情？如果没有，继续沉默。

**类比**：就像一个值班护士——她每隔 15 分钟巡视一次病房（tick），但不是每次巡视都叫醒医生（发言）。只有发现异常指标时才打电话。

## BRIEF_PROACTIVE_SECTION：后台运行时的信息密度调整

主动模式有一个微妙的提示词工程问题：**同一个系统提示词，前台运行和后台运行需要不同的信息密度**。

前台运行时，用户在看着屏幕，代理可以输出详细状态。后台运行时，代理需要压缩信息——因为下一次用户回来时，需要的是**快速摘要**，不是 100 行日志。

```typescript
// 根据是否处于主动模式，调整状态信息格式
BRIEF_PROACTIVE_SECTION  // 后台运行时的状态格式
// vs
标准 BRIEF 格式          // 前台运行时的状态格式
```

**源码**：cli.js ~line 11915

这不是简单地"写短一点"。它意味着：
- 后台模式省略交互性指导（没人看）
- 后台模式强调状态持久化（下次回来要能接上）
- 后台模式调整错误处理策略（不能等用户确认）

| 维度 | 前台模式 | 后台模式 |
|------|---------|---------|
| 输出风格 | 交互式，等待用户输入 | 自主决策，记录日志 |
| 信息密度 | 详细，可追问 | 压缩，一次性完整 |
| 错误处理 | 询问用户 | 尝试修复，记录失败 |
| 上下文保留 | 实时更新 | 关键节点快照 |

## 提示词挑战：有用 vs 不烦人的平衡

主动模式最核心的提示词工程问题是：**如何在 prompt 中定义"值得打断用户"的边界？**

### 理论上的理想指令

```
当检测到重要问题时，通知用户。
```

这句话模型完全能理解——但问题是，"重要"是一个没有客观标准的词。编译失败重要吗？测试挂了 1 个重要吗？依赖冲突重要吗？

### 实际的工程解法

Claude Code 的做法不是在 prompt 中定义"重要"，而是通过**架构分离**来解决：

1. **频率控制在代码层**：`minTickIntervalMinutes` 硬性限制检查频率，prompt 不需要操心"多久查一次"
2. **触发条件在 Hook 层**：PreToolUse / PostToolUse hooks 可以在特定事件发生时才触发检查
3. **判断逻辑在模型层**：最终"要不要打断"由模型根据上下文判断

```
代码层：控制"什么时候可以考虑说话"（频率门控）
Hook 层：控制"什么事件值得考虑说话"（事件门控）
Prompt 层：控制"说话时说什么"（内容门控）
```

**类比**：就像公司的汇报制度——制度规定"每周一开会"（代码层），但只有出了问题才叫老板（Hook 层），叫老板时说什么由你判断（Prompt 层）。三层各管各的，不把所有判断都塞进一个"开会规则"里。

## autoEnable：静默启动的权限问题

主动模式支持 `autoEnable` 配置——启动时自动激活，不需要用户手动打开。

```typescript
// autoEnable 的判断逻辑
if (isEntitled && config.autoEnable) {
  // 自动进入主动模式
  isProactiveActive = true
}
```

这个设计背后有一个提示词哲学问题：**AI 应不应该在用户没明确同意的情况下主动发起对话？**

Claude Code 的回答是分层的：
- GrowthBook flag 确保只有被选中的用户能看到这个功能
- ZDR 检查确保合规客户不会被意外覆盖
- Managed Settings 确保企业可以统一管理
- 最终，`autoEnable` 只是省去了用户手动点击的步骤，**前提是所有门控都已通过**

## 提示词工程技巧总结

| 技巧 | 具体做法 | 效果 |
|------|---------|------|
| 多层门控 | 5 层条件嵌套控制功能可用性 | 灰度+合规+管控三合一 |
| 心跳定时器 | Tick + minTickIntervalMinutes | 控制检查频率而非发言频率 |
| 场景化密度 | BRIEF_PROACTIVE_SECTION 不同格式 | 后台模式自动压缩信息 |
| 架构分离 | 频率/触发/内容三层各司其职 | 不把所有判断塞进 prompt |
| 静默激活 | autoEnable + 门控前置 | 减少用户操作但保留安全 |

## 给抖音文案的启发

> "Claude 学会了'主动汇报'——不是你问它才说，它会在后台盯着，发现问题自己来找你。但它学得很克制：不是每隔几分钟就烦你，而是只有真正重要的事才开口。这就像一个好员工——不会大事小事都问老板，但关键问题绝不隐瞒。"

## 关键洞察

主动模式把提示词工程从"回答问题"升级到"判断时机"——最难的不是"说什么"，而是"什么时候该说"。
