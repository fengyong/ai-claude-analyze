# 技巧二十四：会话状态自愈

> 核心思想：工具调用断裂了？别崩溃——先注入占位结果把对话救回来

---

## 问题本质

Anthropic API 有一条铁律：**每一个 `tool_use` 必须对应一个 `tool_result`**。

就像函数调用必须有返回值一样。如果模型发出了 `tool_use`（比如 "请读取 config.json"），但你没有返回对应的 `tool_result`，API 直接报错，整个对话废掉。

但在真实场景中，不匹配经常发生：

```
场景 1：工具执行崩溃
  → 模型发了 tool_use: Bash("npm install")
  → Bash 进程被 OOM killer 干掉
  → 没有 tool_result 返回
  → API 调用失败

场景 2：流式传输中断
  → 模型并行调用 3 个工具
  → 第 2 个工具的 result 还没到
  → 但第 1 和第 3 个的 result 已经在消息里了
  → 消息序列不连续，配对关系断裂

场景 3：用户手动取消
  → 用户在工具执行中途按了 Ctrl+C
  → tool_use 已经发出，tool_result 没有
  → 下一轮 API 调用必然失败
```

**如果每次不匹配都报错，用户体验就是灾难。**

## 第二层：自愈机制——检测不匹配并注入占位结果

**源码**：pairing ~line 108208

### 核心逻辑

```typescript
// 遍历消息列表，检测 tool_use / tool_result 配对
for (const message of messages) {
  for (const content of message.content) {
    if (content.type === 'tool_use') {
      pendingToolUses.push(content.id)
    }
    if (content.type === 'tool_result') {
      const index = pendingToolUses.indexOf(content.tool_use_id)
      if (index !== -1) {
        pendingToolUses.splice(index, 1)  // 配对成功，移除
      }
    }
  }
}

// 配对失败的 tool_use → 注入占位结果
for (const orphanId of pendingToolUses) {
  injectPlaceholderResult(orphanId, {
    content: "[Tool result missing — auto-injected placeholder]",
    is_error: true
  })
}
```

### 注入的占位结果长什么样

```json
{
  "type": "tool_result",
  "tool_use_id": "orphan_tool_use_123",
  "content": [
    {
      "type": "text",
      "text": "[Tool result missing — auto-injected placeholder]"
    }
  ],
  "is_error": true
}
```

**关键细节**：
- `is_error: true`——明确告诉模型"这不是真实结果，是占位符"
- 模型看到这个会知道工具执行失败了，可以自行决定重试或换策略
- 比 API 直接报错好一万倍——至少对话还能继续

## 第三层：strictToolResultPairing——什么时候应该让错误暴露

并非所有场景都适合自动修复。`strictToolResultPairing` 模式提供了"严格模式"：

```typescript
// 严格模式：不匹配直接抛错，不自动修复
if (config.strictToolResultPairing) {
  throw new Error(
    `Tool use ${orphanId} has no matching tool result. ` +
    `Strict pairing mode is enabled.`
  )
}

// 宽松模式（默认）：自动注入占位结果
else {
  injectPlaceholderResult(orphanId)
  telemetry.track('tengu_tool_result_pairing_repaired')
}
```

### 什么时候用严格模式？

| 场景 | 推荐模式 | 原因 |
|------|---------|------|
| 生产环境交互 | 宽松（默认） | 用户体验优先，不能因为一个工具失败就崩 |
| 调试/开发 | 严格 | 发现潜在的工具编排 bug |
| CI/CD 管道 | 严格 | 宁可失败也不能静默跳过 |
| 长时间自主任务 | 宽松 | 任务不能因为偶发失败中断 |

### 遥测追踪

每次自动修复都会触发遥测事件：

```typescript
telemetry.track('tengu_tool_result_pairing_repaired', {
  tool_use_id: orphanId,
  tool_name: orphanTool.name,
  message_index: orphanMessageIndex,
  repair_type: 'placeholder_injection'
})
```

**这意味着**：Anthropic 在追踪"修复频率"。如果某个工具的配对失败率异常高，他们可以针对性地改进那个工具的稳定性。

## 第四层：隐藏陷阱——20 个内容块的缓存悬崖

**源码**：cache breakpoint ~line 152975

这是一个不为人知但影响巨大的细节：

### 缓存断点的回溯限制

```
Anthropic API 的 prompt cache 工作方式：
  → 在消息序列中标记"缓存断点"
  → 下一次请求时，API 从断点位置开始匹配缓存
  → 匹配成功 → 缓存命中（省钱）
  → 匹配失败 → 缓存未命中（全价）

Claude Code 的缓存断点策略：
  → 每隔一定间隔插入 __CACHE_BREAKPOINT__ 标记
  → 回溯查找匹配时，最多向后看 20 个 content blocks
```

### 为什么 20 个 block 是个问题

```
正常对话（每轮 2-3 个 content blocks）：
  Round 1: [assistant: text] [assistant: tool_use] [user: tool_result]  ← 3 blocks
  Round 2: [assistant: text] [assistant: tool_use] [user: tool_result]  ← 3 blocks
  ...
  20 blocks ≈ 6-7 轮对话 → 缓存断点能覆盖 → ✅

Agentic 循环（每轮 10+ 个 content blocks）：
  Round 1: [text] [tool_use×4] [tool_result×4] [text]  ← 10 blocks
  Round 2: [text] [tool_use×4] [tool_result×4] [text]  ← 10 blocks
  ...
  20 blocks ≈ 2 轮对话 → 缓存断点覆盖不住 → ❌

结果：
  → 第 3 轮开始，缓存断点找不到匹配
  → 静默缓存未命中（silent cache miss）
  → 每次请求都按全价计费
  → 用户无感知，但费用翻倍甚至更多
```

### 这对提示词工程意味着什么

```
如果你的 Agent 循环每轮产生大量工具调用：

  好的情况：每轮 2-3 个工具 → 缓存友好 → 成本可控
  坏的情况：每轮 10+ 个工具 → 缓存悬崖 → 成本飙升

应对策略：
  1. 减少并行工具调用数量（分批执行）
  2. 让 Agent 在工具调用之间穿插文本回复（增加 content block 密度）
  3. 监控 cache_read_input_tokens 比例，发现异常及时调整
```

## 提示词工程技巧总结

| 技巧 | 具体做法 | 效果 |
|------|---------|------|
| 自愈注入 | 检测不匹配 → 注入占位 tool_result | 对话不中断 |
| 严格模式 | strictToolResultPairing 可选 | 调试时暴露问题 |
| 遥测追踪 | 记录每次修复事件 | 量化工具稳定性 |
| 缓存悬崖 | 20 block 回溯限制 | Agent 循环需控制工具密度 |
| is_error 标记 | 占位结果标记为错误 | 模型知道这不是真实结果 |

---

## 关键洞察

> **会话状态的完整性比"完美真实"更重要——先活下来再说。一个注入的占位结果告诉模型"这里出错了"，比整条对话报废好一万倍。但别忘了隐藏的 20 block 缓存悬崖：你的 Agent 每轮调用太多工具，不仅费钱，还会让 prompt cache 彻底失效。**
