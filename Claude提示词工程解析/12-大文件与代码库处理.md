# 大模型如何处理大量文件和代码？

> 核心发现：没有 RAG，没有向量搜索，没有代码索引——用的是"人类式的笨办法"

---

## 用户的问题

"我有很多文件很多行代码，大模型是如何指导这些文件里面的内容，帮我做下一步决策的？"

---

## 直觉回答 vs 真实机制

**直觉**：大模型一定先把整个代码库"读完"建立索引，然后才能回答问题。

**现实**：Claude Code **没有**做任何预处理。没有索引，没有 RAG，没有向量搜索。它从零开始，像一个刚接手项目的程序员——先搜，再读，再判断。

---

## 处理流程：先搜再读

```
用户："这个项目的认证逻辑在哪里？"

第一步：GlobTool → "找所有 auth 相关的文件"
        → src/auth/login.ts, src/middleware/auth.ts, src/utils/token.ts

第二步：GrepTool → "搜 'authenticate' 关键词出现在哪些文件"
        → src/auth/login.ts:42, src/middleware/auth.ts:18

第三步：FileReadTool → "读这两个文件"
        → 读取 src/auth/login.ts 第 1-200 行
        → 读取 src/middleware/auth.ts 第 1-150 行

第四步：模型判断 → "认证逻辑在 login.ts 的 authenticateUser 函数中"
```

**关键**：模型每次只读它**当前需要**的部分，不是一次性读完整个代码库。

---

## 文件阅读的三道硬限制

### 限制 1：文件大小 ≤ 256 KB

```
maxSizeBytes = 256 * 1024  // 256 KB
```

超过 256 KB 的文件直接拒绝读取，必须用 offset/limit 分段读。

### 限制 2：单次读取 ≤ 2000 行

```
MAX_LINES_TO_READ = 2000
```

一个 5000 行的文件，必须分 3 次读：
```
Read(file, offset=1, limit=2000)    → 第 1-2000 行
Read(file, offset=2001, limit=2000) → 第 2001-4000 行
Read(file, offset=4001, limit=2000) → 第 4001-5000 行
```

### 限制 3：Token 预算 ≤ 25,000 tokens

```
maxTokens = 25,000
```

即使文件在 256 KB / 2000 行以内，如果内容的 token 数超过 25000（大约 10 万字符），也会被截断。

**三层限制叠加**：实际能读到的内容远小于上下文窗口的 200K tokens。

## 搜索工具的限制

### GlobTool（文件名搜索）

| 参数 | 值 |
|------|---|
| 最大结果数 | 100 个文件 |
| 最大字符数 | 100,000 字符 |
| 排序 | 修改时间（最新的优先） |
| 底层引擎 | ripgrep |

```
Glob("**/*.ts")  → 返回最多 100 个 .ts 文件路径
```

### GrepTool（内容搜索）

| 参数 | 值 |
|------|---|
| 最大结果数 | 250 行匹配 |
| 最大字符数 | 20,000 字符 |
| 每行最大列数 | 500 字符（防止 base64/压缩代码污染输出） |
| 排除目录 | .git, .svn, .hg, .bzr |

```
Grep("authenticate")  → 返回最多 250 行包含 "authenticate" 的代码
```

**限制意味着**：如果一个关键词在 250 个以上位置出现，模型看不到后面的部分。

---

## 工具返回结果的全局限制

```
DEFAULT_MAX_RESULT_SIZE_CHARS = 50,000    // 单个工具结果上限
MAX_TOOL_RESULT_TOKENS = 100,000          // 单个工具结果 token 上限
MAX_TOOL_RESULTS_PER_MESSAGE_CHARS = 200,000  // 一轮对话中所有工具结果总和上限
```

**关键**：即使模型同时调用了 4 个工具（并行），所有返回结果的总和也不能超过 200,000 字符。这大约是 50K tokens——上下文窗口的 1/4。

---

## 去重机制：不重复读已读过的文件

```
如果：
  - 同一个文件的同一行范围被再次读取
  - 文件的 mtime（修改时间）没变
则：
  → 返回 "File unchanged since last read"
  → 不消耗 token
```

**效果**：重复读同一段代码不会浪费上下文窗口。

## 上下文压缩：200K 窗口的生存策略

### 自动压缩触发条件

```
MODEL_CONTEXT_WINDOW_DEFAULT = 200,000 tokens
AUTOCOMPACT_BUFFER_TOKENS = 13,000 tokens

当已用 tokens > (200,000 - 13,000) = 187,000 时
→ 触发自动压缩
```

### 压缩过程

```
压缩前的对话：
  [系统 prompt] [用户消息1] [模型回复1] [工具结果1] [模型回复2] ...
  → 总共 187,000 tokens

压缩后：
  [系统 prompt] [九段式摘要] [最近5个文件内容]
  → 总共约 20,000-50,000 tokens

九段式摘要结构：
  1. Primary Request and Intent（主要请求和意图）
  2. Key Technical Concepts（关键技术概念）
  3. Files and Code Sections（文件和代码）
  4. Errors and Fixes（错误和修复）
  5. Problem Solving（问题解决过程）
  6. All User Messages（所有用户消息）
  7. Pending Tasks（待完成任务）
  8. Current Work（当前工作）
  9. Optional Next Step（下一步建议）
```

### 压缩后的文件恢复

压缩后，最多恢复 **5 个最近读取的文件**，预算 **50K tokens**，每个文件最多 **5K tokens**。

```
压缩前：模型读了 20 个文件
压缩后：只保留最近 5 个文件的摘要内容，其余丢失
```

**这意味着**：长对话中，模型会"忘记"早期读过的文件内容，需要重新读取。

---

## 微压缩：延迟全面压缩的技巧

在触发全面压缩之前，有两种"微压缩"策略：

### 缓存微压缩

```
使用 API 的缓存编辑功能，移除旧的工具返回结果
但不破坏缓存前缀（静态 prompt 的缓存仍然有效）
```

### 时间微压缩

```
如果距上一条助手消息的时间超过阈值：
→ 将旧工具结果替换为 "内容已清除"
→ 保留消息结构，释放 token 空间
```

**可压缩的工具**：Read, Bash/Grep/Glob, WebSearch/WebFetch, Edit/Write

## Explore Agent：用便宜模型先探路

```
Explore Agent:
  - 模型：haiku（外部用户）/ inherit（内部用户）
  - 权限：只读（不能编辑、写入、调用其他 Agent）
  - 用途：快速探索代码库
  - 最少查询数：3
  - 跳过 CLAUDE.md（为了速度）
```

**用法**：主 Agent 派一个"侦察兵"出去找文件，侦察兵返回摘要，主 Agent 根据摘要决定读哪些文件。

```
主 Agent："认证逻辑在哪里？"
  ↓
Explore Agent（haiku，便宜）：
  → Glob("**/auth*")  → 找到 5 个文件
  → Grep("authenticate")  → 定位到 3 个关键函数
  → Read(关键文件片段)  → 理解大致结构
  → 返回摘要："认证逻辑在 src/auth/login.ts 的 authenticateUser 函数中"
  ↓
主 Agent：读取 src/auth/login.ts，做具体决策
```

**本质**：用便宜模型做粗筛，用贵模型做精读。

## LSP：唯一接近"代码索引"的工具

```
LSPTool 操作：
  - goToDefinition      → 跳转到定义
  - findReferences      → 查找引用
  - hover               → 查看类型信息
  - documentSymbol      → 文件内符号列表
  - workspaceSymbol     → 工作区符号搜索
  - goToImplementation  → 跳转到实现
  - callHierarchy       → 调用层级
```

**但**：LSP 依赖外部 LSP 服务器的运行状态，不是 Claude Code 自身的能力。而且 LSP 结果也有大小限制（100,000 字符）。

## 没有的东西

| 常见假设 | 实际情况 |
|---------|---------|
| "先读完整个代码库建索引" | 没有，每次只读需要的部分 |
| "用 RAG 检索相关代码" | 没有，代码库中没有向量搜索 |
| "用 Tree-sitter 解析代码结构" | 只有 Bash 用了，其他语言没有 |
| "有代码索引系统" | 没有，只是检测外部索引工具是否存在 |
| "读过的文件都记住" | 压缩后只保留最近 5 个 |

## 完整的工作流程图

```
用户问题
    │
    ▼
模型判断："我需要看哪些文件？"
    │
    ├──→ GlobTool：按文件名模式找文件（最多 100 个）
    ├──→ GrepTool：按关键词搜内容（最多 250 行）
    │
    ▼
模型判断："这些搜索结果中哪些最重要？"
    │
    ├──→ FileReadTool：读取关键文件（最多 2000 行/次，25K tokens/次）
    │    └── 大文件分段读取（offset/limit）
    │
    ▼
模型判断："基于这些代码，我理解了问题，现在做决策"
    │
    ├──→ EditTool：修改代码
    ├──→ BashTool：运行测试
    ├──→ 再次 Grep/FileRead：验证理解
    │
    ▼
上下文快满时（187K tokens）：
    │
    ├──→ 自动压缩：九段式摘要 + 保留最近 5 个文件
    │
    ▼
压缩后继续工作（重新读取需要的文件）
```

## 给抖音文案的启发

> "你以为 Claude 读完了你整个项目？它没有。它每次最多读 2000 行，而且读完的文件过一会就忘了——对话太长时它会自动压缩，只保留最近 5 个文件。它像一个很聪明但注意力有限的程序员：先搜关键词，再读关键文件，读完就判断。整个过程没有索引，没有缓存，没有预处理——每次都是从零开始。"
