# 技巧四：工具优先级 & 防呆设计

> 核心思想：每个工具的 prompt 不只说"怎么用"，更说"什么时候不该用"

---

## 工具 Prompt 的共性模式

36 个工具 prompt 中，反复出现以下模式：

### 模式 1：工具偏好层级

几乎所有工具 prompt 都在强调同一个层级：

```
专用工具 > Bash > 什么都不做
```

具体例子：

| 场景 | 应该用 | 不应该用 |
|------|--------|---------|
| 读文件 | `Read` 工具 | `cat`, `head`, `tail` |
| 搜内容 | `Grep` 工具 | `grep`, `rg` in Bash |
| 找文件 | `Glob` 工具 | `find`, `ls` |
| 写文件 | `Write` 工具 | `echo`, heredoc |
| 编辑文件 | `Edit` 工具 | `sed`, `awk` |

**Why**: 专用工具提供更好的错误信息、权限检查和用户体验。

### 模式 2：前置条件门控

工具 prompt 通过文字强制前置操作：

**FileWriteTool**：
> "If this is an existing file, you MUST use the Read tool first. This tool will fail if you did not read the file first."

**FileEditTool**：
> "You must use the Read tool at least once before editing."

这不是软建议——工具会**实际检查**并拒绝执行，形成硬依赖。

### 模式 3：When to Use / When NOT to Use 决策树

多个工具使用结构化的决策框架：

**EnterPlanModeTool**：
```
When to Use (7 个条件):
- New Feature Implementation
- Multiple Valid Approaches
- Code Modifications
- Architectural Decisions
- Multi-File Changes
- Unclear Requirements
- User Preferences Matter

When NOT to Use (4 个条件):
- Single-line fixes
- Adding a single function
- Very specific instructions
- Pure research/exploration
```

**TaskCreateTool**：
```
When to Use: 3+ distinct steps, non-trivial tasks, user requests
When NOT to Use: single task, trivial, < 3 steps
```

### 模式 4：反模式警告

工具 prompt 不只教正确用法，还明确指出常见错误：

**ExitPlanModeTool**：
> "Do NOT use AskUserQuestion to ask 'Is my plan ready?' — use ExitPlanMode instead."

**AskUserQuestionTool**：
> "Do not ask 'Is my plan okay?' — that is what ExitPlanMode does."

**BashTool**：
> "NEVER run `git commit --amend`" / "NEVER skip hooks (`--no-verify`)" / "NEVER force push"

## 精选工具 Prompt 深度分析

### BashTool（~370 行，最复杂的工具 prompt）

**Git 安全协议**：
```
1. NEVER update git config
2. NEVER run destructive git commands unless user explicitly asks
3. NEVER skip hooks (--no-verify)
4. ALWAYS create NEW commits rather than amending
5. NEVER push to remote unless explicitly asked
```

**PR 创建四步流程**（并行优化）：
```
Step 1: git status + git diff + git log  ← 并行
Step 2: 分析变更，草拟 PR 标题和描述
Step 3: create branch + push + gh pr create  ← 并行
Step 4: git status 验证
```

**Sleep/监控规则**：
- 不要在命令之间 `sleep`
- 不要用 sleep 轮询外部进程
- 用 `run_in_background` 处理长时间任务
- 只有必须轮询时才用短暂 sleep（1-5 秒）

### GrepTool —— 工具名动态插值

```typescript
// 使用常量引用工具名，避免硬编码
`ALWAYS use ${GREP_TOOL_NAME} over ${BASH_TOOL_NAME}`
```

**Why**: 如果工具改名，只需改一个常量，所有引用自动更新。

### SkillTool —— 上下文预算系统

```typescript
SKILL_BUDGET_CONTEXT_PERCENT = 0.01  // 1% 的上下文窗口
```

技能列表被动态截断以适应 1% 的预算：
- 捆绑技能优先获得完整描述
- 自定义技能可能只显示名称
- `formatCommandsWithinBudget()` 负责裁剪

**Why**: 技能列表可能很长，但不应该占用太多上下文窗口。

### ScheduleCronTool —— 反雷群效应

```
"Avoid the :00 and :30 minute marks when the task allows it"
"Every user who asks for '9am' gets 0 9, and every user who asks for
 'hourly' gets 0 * — which means requests from across the planet land
 on the API at the same instant."
```

**Why**: 所有人都选整点执行 → API 同时收到大量请求 → 雷群效应（thundering herd）。通过偏移几分钟分散负载。

## 提示词工程技巧总结

| 技巧 | 具体做法 | 效果 |
|------|---------|------|
| 前置条件门控 | "You MUST do X first, or this tool will fail" | 硬依赖，不是软建议 |
| 决策树 | "When to Use" + "When NOT to Use" | 消除模糊地带 |
| 反模式清单 | 明确列出 "NEVER do X" | 防止常见错误 |
| 工具名常量化 | `${TOOL_NAME}` 而非硬编码 | 改名时自动同步 |
| 上下文预算 | 给不重要内容设 token 预算上限 | 保护核心上下文窗口 |
| 反雷群效应 | 人为制造随机偏移 | 分散系统负载 |
