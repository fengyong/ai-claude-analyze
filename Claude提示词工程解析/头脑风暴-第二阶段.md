# 头脑风暴第二阶段：Claude 如何用提示词掌控大模型

---

## 大模型的本质

**常见误解**：大模型掌握了人类一切文字知识，是一种全能力量。

**更精确的理解**：
- 它掌握了**语言的统计规律**，不是"知识"
- 它不知道"什么是真的"，只知道"什么话在人类文本中经常出现"
- 它像一个**读了全天下书但从没出过门的人**——知识渊博，但没有直觉、没有常识、没有意志

**核心比喻**：
> 大模型是一条大河。它有无穷的水量（能力），但水会自然流向最低处（训练数据中概率最高的方向）。Prompt 就是河堤——不产生水，但决定水往哪里流。

---

## 逐步分析：Claude 用提示词掌控大模型的七步

### 第一步：身份锚定 — "你是谁"

```
"You are Claude, an AI assistant made by Anthropic."
"You are the official CLI for Claude."
```

**作用机制**：大模型没有"自我"。它在每次对话开始时需要被"初始化"——告诉它应该以什么身份回应。

**类比**：就像给一个演员剧本的第一页——"你扮演一个外科医生"。演员（大模型）本身什么角色都能演，但必须先知道演谁。

**实际效果**：没有这个锚定，模型可能以任何风格回应——有时像客服，有时像诗人，有时像论坛网友。

---

### 第二步：安全围栏 — "你不能做什么"

```
"Assist with authorized security testing..."
"Refuse requests for destructive techniques, DoS attacks..."
"Dual-use tools require clear authorization context."
```

**作用机制**：大模型的能力没有天然边界。它能写攻击代码，和写防御代码一样轻松。安全指令是在能力前面加一道门。

**关键细节**：安全指令被放在 prompt 的最前面（静态层），因为 transformer 的注意力机制会让前面的内容权重更高。

**类比**：就像给一把万能钥匙加上使用规则——钥匙能开所有门，但规则告诉你哪些门不能开。

---

### 第三步：行为塑造 — "你该怎么做事"

```
"Don't over-engineer. Three similar lines of code is better than a premature abstraction."
"Be careful not to introduce security vulnerabilities."
"NEVER skip hooks. NEVER force push."
```

**作用机制**：大模型默认会生成"看起来最可能正确"的代码——但"最可能"不等于"最好"。行为准则通过否定形式（Don't / NEVER）重新定义"最好"。

**为什么否定比肯定有效**：
- 肯定："写简洁的代码" → 模型不确定什么算简洁
- 否定："不要添加用不到的错误处理" → 明确的边界

**类比**：就像教孩子骑自行车——"看着前方"是建议，"别看脚下"是救命规则。

---

### 第四步：工具引导 — "你该用什么"

```
"Use Read tool over cat/head/tail"
"Use Grep tool over grep/rg in Bash"
"Reserve Bash for system commands"
```

**作用机制**：大模型在训练数据中见过无数 `cat file.txt` 的用法，自然会倾向使用它。工具引导通过显式覆盖训练数据中的默认偏好，强制模型使用更优工具。

**三层策略**：
1. **Prompt 层**：文字告诉你用什么
2. **工具描述层**：每个工具有自己的 prompt.ts
3. **工具层**：物理上只提供应该用的工具

**类比**：就像给一个什么都会用的老工匠一套专用工具——他能用石头砸钉子，但你给了他锤子，他会用得更好。

---

### 第五步：输出控制 — "你该怎么说"

```
"Go straight to the point."
"Don't use emojis unless the user explicitly requests it."
"Output in file_path:line_number format."
```

**作用机制**：大模型的默认输出风格是"礼貌而冗长"——因为训练数据中礼貌的文本更多。输出控制覆盖这个默认值。

**进阶技巧**：Output Style 覆盖层可以在运行时切换风格——同一模型，在"解释模式"下加知识块，在"学习模式"下出题给用户。

**类比**：就像同一个演员，可以演喜剧也可以演悲剧——导演（prompt）决定了表演风格。

---

### 第六步：上下文管理 — "你该记住什么"

```
九段式摘要：Primary Request → Key Concepts → Files → Errors → ...
会话记忆：十段式模板，Token 预算 12000
```

**作用机制**：大模型的上下文窗口有限。长对话会溢出。上下文管理通过压缩和结构化记忆让模型在有限窗口内保持一致性。

**核心洞察**：这不是"教"模型怎么做，而是给模型一个格式，让它按格式输出——模型天生擅长"按格式填空"。

**类比**：就像给一个记忆力差但很聪明的人一个笔记本模板——他不会自己记笔记，但给他模板他就能填得很好。

---

### 第七步：子 Agent 编排 — "你该派谁去做"

```
"Launch multiple agents in parallel when possible"
"Brief the agent like a smart colleague who just walked into the room"
"Never delegate understanding"
```

**作用机制**：一个大模型实例的能力是固定的。但通过分发子任务给专门化的 Agent（探索、验证、规划），可以把固定能力扩展为专业化的协作系统。

**关键设计**：
- 探索 Agent 只能看不能改（工具层物理限制）
- 验证 Agent 必须尝试破坏（prompt 层对抗性指令）
- 规划 Agent 只能读不能写（工具层 + prompt 层双重限制）

**类比**：就像一个将军——他本人什么都能做，但更有效的方式是派侦察兵、参谋、工兵各司其职。

---

## 提示词工程的本质

```
大模型 = 无边界的能量
提示词 = 能量的容器

没有容器，能量是混乱的（大模型的默认输出）
有了容器，能量是可控的（Claude Code 的精准行为）
```

**Claude 的提示词工程做了一件事**：把一个"什么都会但不知道该做什么"的大模型，变成一个"知道该做什么、怎么做、用什么做"的工程系统。

## 大模型的本质：无状态的失忆天才

### 用户的关键洞察

> 大模型自身记住了无穷的知识，但不记忆用户的任何信息，无动于衷。每次对话都是带着历史去全新推理。所以一次对话可以更换不同的模型也能保持连续。

### 精确化

**大模型是一个冻结的函数**：

```
f(prompt) → response
```

- 不会因为和你聊过而改变
- 第一次对话和第一万次对话，调用的是完全同一个模型
- **它不记得你**

**"带着历史全新推理"——完全正确**：

每次 API 调用，发送的是完整历史的拷贝：

```
[
  { "role": "system",   "content": "系统 prompt..." },
  { "role": "user",     "content": "第一条消息" },
  { "role": "assistant","content": "第一条回复" },
  { "role": "user",     "content": "第二条消息" },
  ...
]
```

模型收到的是完整历史，然后从零开始重新推理。它不是"回忆"上一轮说了什么，而是**重新阅读**你说过的一切。

**换模型保持连续——对，但有条件**：
- 历史消息是**文本**，不依赖任何特定模型
- 新模型拿到同样的历史，从头推理
- 就像换了一个读者读同一本书，故事本身没变

**但有条件**：
- 不同模型的"能力"不同（换弱模型可能读不懂复杂上下文）
- 不同模型的"风格"不同（同样的历史，回复风格可能变化）
- System prompt 如果有模型特定内容，换模型时可能需要调整

### 终极比喻

```
大模型 = 一个失忆的天才
         - 脑子里装着全人类的知识（训练数据）
         - 但不记得任何一次对话
         - 每次对话都是"第一次见面"
         - 你把之前的聊天记录递给他，他读完才"想起来"

上下文 = 递给他看的聊天记录
         - 他不是回忆，是重新读
         - 记录太长就看不完了（上下文窗口限制）
         - 你换一个人（模型）给他看同样的记录，他也能接上

提示词 = 你递给他的聊天记录的第一部分
         - 告诉他是谁、该做什么、不该做什么
         - 这部分不包含你的对话历史，但决定他怎么处理你的历史
```

### 一句话总结

> **大模型是无状态的函数。所有"记忆"都是外部传入的文本。提示词工程就是管理这些文本的结构和内容。**

这也是为什么 Claude Code 的 prompt 工程这么复杂——因为要在一个"失忆的天才"身上，构建出看起来像"有记忆的助手"的体验。
