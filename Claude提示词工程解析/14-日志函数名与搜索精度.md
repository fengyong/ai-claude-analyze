# 日志带函数名：Grep 搜索的泛滥与精准

> 核心发现：不是"泛滥 vs 精准"的二选一，而是"泛滥让粗搜变差，但让精搜变好"

---

## 问题

AI 生成的代码倾向于把函数名带入日志：

```typescript
logger.debug('authenticateUser: starting auth check')
logger.info('authenticateUser: token validated')
logger.error('authenticateUser: failed', { userId })
```

GrepTool 限制 250 行匹配。这种模式会让关键词泛滥，还是更容易定位？

---

## Grep 限制回顾

| 参数 | 值 |
|------|---|
| 最大结果数 | 250 行匹配 |
| 最大字符数 | 20,000 字符 |
| 每行最大列数 | 500 字符 |

如果 `authenticate` 在 300 个位置出现，模型只能看到前 250 行。

---

## 两种场景对比

### 场景 A：日志也用函数名

```
函数定义：function authenticateUser() { ... }
日志：    logger.debug('authenticateUser: starting auth check')
日志：    logger.info('authenticateUser: token validated')
日志：    logger.error('authenticateUser: failed', { userId })
调用处：  authenticateUser(req, res)

Grep("authenticate") → 所有这些行都匹配 → 可能超过 250 行
```

### 场景 B：日志用业务描述

```
函数定义：function authenticateUser() { ... }
日志：    logger.debug('starting auth check')
日志：    logger.info('token validated')
日志：    logger.error('auth failed', { userId })
调用处：  authenticateUser(req, res)

Grep("authenticate") → 只有函数定义和调用处匹配 → 结果少但精准
```

---

## 但场景 A 有更好的精搜路径

日志带函数名的好处是支持**渐进式搜索**：

```
第一步：Grep("authenticate")
        → 返回 300 行，被截断到 250 行
        → 模型发现"太多了"

第二步：Grep("authenticateUser: failed")
        → 精确匹配，返回 3-5 行
        → 直接定位到失败位置
```

场景 B 中，精搜只能搜 `auth failed`，但这不一定唯一——可能有 `handleAuthFailed`、`onAuthFailure` 等同义表达。

---

## 模型的渐进搜索策略

```
粗搜（宽泛关键词）
  │
  ├── 结果 < 50 行 → 直接读文件，精确定位
  │
  ├── 结果 50-250 行 → 够用，选择关键文件读取
  │
  └── 结果 > 250 行 → 截断，模型意识到"太泛了"
       │
       ├── 缩小搜索范围：Grep("authenticate", path="src/auth/")
       │
       ├── 增加特异性：Grep("authenticateUser: failed")
       │
       └── 换工具：Glob("**/auth*.ts") → 读文件
```

日志带函数名恰好支持第三种策略——模型可以从 `authenticate` 缩小到 `authenticateUser: failed`，搜索精度从 300 行降到 3-5 行。

---

## GrepTool 的 500 字符行限制也有影响

```typescript
// 如果日志很长：
logger.debug('authenticateUser: checking token for user with very long description...')

// 超过 500 字符的行被截断
// 但函数名通常在行首，所以截断不影响定位
```

---

## 实际最优策略

| 做法 | 粗搜效果 | 精搜效果 | 综合评价 |
|------|---------|---------|---------|
| 日志带完整函数名 | 泛滥 | 精确 | 支持渐进搜索 |
| 日志带文件路径+函数名 | 更泛滥 | 最精确 | 粗搜几乎不可用 |
| 日志只写业务描述 | 不泛滥 | 不精确 | 粗搜好，精搜差 |
| 日志不带任何标识 | 不泛滥 | 需要读文件 | 定位成本高 |

**最优解**：日志带函数名。泛滥是粗搜的问题，但模型会自动从粗搜过渡到精搜。函数名提供了天然的"精确搜索锚点"。

---

## 核心结论

> **日志带函数名不是"关键词泛滥"的问题，而是"搜索粒度可调"的优势。**
>
> 粗搜 `authenticate` 返回 300 行 → 模型知道"太泛了"
> 精搜 `authenticateUser: failed` 返回 3 行 → 精确定位
>
> 如果日志不带函数名，粗搜只返回 10 行 → 模型以为找到了，但其实错过了其他位置。
>
> **泛滥是信号，不是噪音——它告诉模型"你需要更具体的搜索"。**

---

## 给抖音文案的启发

> "AI 写的日志里总带着函数名，你觉得冗余？不，这是给搜索引擎留的精确锚点。Claude 搜 'authenticate' 会返回 300 行——太多了。但它会自动缩小到 'authenticateUser: failed'，3 行精确定位。关键词泛滥不是问题，问题是你的搜索够不够聪明。"
