# 技巧二十二：邮箱式 IPC——多代理的消息传递

> 核心思想：一个代理上下文不够？组个团队，用"邮箱"协调

---

## 问题本质

单个代理的上下文窗口再大，也有两个致命瓶颈：

1. **上下文污染**：探索代码、执行修改、验证结果全挤在一个上下文里，互相干扰
2. **串行瓶颈**：主代理一次只能做一件事，无法同时探索多个方向

就像一个项目经理既要做需求分析、又要写代码、还要跑测试——脑子会炸。

Claude Code 的解法：**Team 集群架构**——主会话作为 Team Lead，按需生成队友（in-process agents），每个队友有独立的上下文窗口和工具权限。

---

## 第一层：集群架构全景

**源码**：Team/swarm ~line 81735, TeamCreate ~line 64330

```
┌────────────────────────────────────────────────────┐
│                  Team Lead（主会话）                 │
│                                                    │
│   - 拥有完整工具权限                                │
│   - 管理共享任务列表                                │
│   - 通过 mailbox 收发消息                           │
│   - 决定何时生成/终止队友                            │
│                                                    │
│   ┌──────────┐  ┌──────────┐  ┌──────────┐        │
│   │ Teammate │  │ Teammate │  │ Teammate │        │
│   │ "探索者"  │  │ "编码者"  │  │ "审查者"  │        │
│   │ 只读     │  │ 完整权限  │  │ 只读     │        │
│   │ 独立上下文│  │ 独立上下文│  │ 独立上下文│        │
│   └──────────┘  └──────────┘  └──────────┘        │
│        ↑              ↑             ↑              │
│        └──────── mailbox ───────────┘              │
│           SendMessage(to: "<name>")                │
└────────────────────────────────────────────────────┘
```

**关键设计**：每个 Teammate 拥有独立的上下文窗口。这意味着探索者的代码搜索结果不会污染编码者的上下文——各自干净，互不干扰。

## 第二层：邮箱式 IPC——SendMessage 的消息路由

### 消息格式

Team 内部通信使用 `teammate-message` XML 标签结构：

```xml
<teammate-message>
  <from>team-lead</from>
  <to>explorer-agent</to>
  <content>
    请探索 src/services/compact/ 目录下的所有文件，
    列出每个文件的职责和关键函数。
  </content>
</teammate-message>
```

### SendMessage 工具路由

```typescript
// 消息路由机制
SendMessage({
  to: "<teammate-name>",    // 目标队友名称
  content: "指令或数据"      // 消息内容
})

// Team Lead 广播给所有队友
SendMessage({
  to: "*",                   // 通配符：所有队友
  content: "阶段性同步信息"
})
```

### 为什么用"邮箱"而不是共享内存？

| 通信方式 | 优点 | 风险 | Claude Code 选择 |
|---------|------|------|----------------|
| 共享内存 | 低延迟 | 状态竞争、上下文污染 | ❌ 不采用 |
| 邮箱式 IPC | 隔离、可追踪 | 略有延迟 | ✅ 采用 |
| 文件传递 | 持久化 | 一致性问题 | 辅助方式 |

**设计哲学**：Go 语言的谚语"Don't communicate by sharing memory; share memory by communicating"在这里完美适用。每个 Teammate 的上下文窗口就是它的"私有内存"，Team Lead 通过邮箱发送的是"指令副本"而非引用。

## 第三层：权限集中——PermissionSync 如何保持安全

**源码**：PermissionSync ~line 81864

分布式代理系统的最大风险：**权限失控**。如果每个 Teammate 都能独立批准工具调用，安全性就成了一盘散沙。

```
Teammate 想执行 Bash("rm -rf /tmp/build")
       │
       ▼
Teammate 没有直接批准权限
       │
       ▼
通过 mailbox 向 Team Lead 发送 PermissionRequest
       │
       ▼
Team Lead 审批（基于全局上下文判断）
       │
       ├── 批准 → PermissionGranted → Teammate 执行
       └── 拒绝 → PermissionDenied  → Teammate 跳过
```

**为什么必须集中审批？**

1. **全局视野**：Lead 知道其他队友在做什么，能判断冲突
2. **安全边界**：用户只和 Lead 交互，信任关系建立在 Lead 上
3. **审计追踪**：所有批准/拒绝记录集中在一个地方

这就像公司的采购审批——部门可以申请，但签字权在财务总监手里。

## 第四层：任务分配——基于代理能力的智能分发

### Agent 类型对比

| 特性 | General-Purpose（通用） | Explore（探索） | Plan（规划） |
|------|----------------------|----------------|-------------|
| 工具权限 | `['*']` 全部工具 | 只读工具 | 只读工具 |
| 文件修改 | 允许 | 禁止 | 禁止 |
| 典型用途 | 编码、执行 | 代码搜索、侦察 | 架构设计、方案规划 |
| 上下文需求 | 高（需要理解全貌） | 低（只看不改） | 中（需要结构化输出） |
| 成本 | 高 | 低（可用 Haiku） | 中 |

### 任务分配逻辑

```
Team Lead 收到任务："重构认证模块并添加测试覆盖"
       │
       ├──→ Plan Agent："分析认证模块的架构，输出重构方案"
       │         只读，输出结构化方案
       │
       ├──→ Explore Agent × 2（并行）：
       │     "搜索所有 import auth 的文件"
       │     "搜索所有测试文件中 auth 相关的 case"
       │         只读，并行执行，快速完成
       │
       └──→ General-Purpose Agent：
             "按方案执行重构"
             完整权限，串行执行
```

**并行 + 分类**：只读代理可以安全并行（没有副作用），写入代理必须串行（避免冲突）。

## 第五层：tmux/iTerm2 渲染——可视化的团队协作

每个 Teammate 在终端中渲染为独立的 pane：

- **每个 pane 有独特的颜色标识**——一眼分辨谁是谁
- **实时显示工具调用和输出**——Team Lead 可以"看到"队友在做什么
- **支持 iTerm2 和 tmux 两种渲染后端**

```
┌─────────────────────┬─────────────────────┐
│  Team Lead          │  Explorer (蓝色)     │
│  > 正在等待结果...    │  > Grep "auth"...   │
│                     │  > 找到 23 个匹配     │
├─────────────────────┼─────────────────────┤
│  Encoder (绿色)      │  Plan Agent (黄色)   │
│  > 编辑 auth.ts...   │  > 分析依赖关系...    │
│  > 写入测试...        │  > 输出重构方案       │
└─────────────────────┴─────────────────────┘
```

## 提示词工程技巧总结

| 技巧 | 具体做法 | 效果 |
|------|---------|------|
| 独立上下文 | 每个 Teammate 独立 context window | 避免上下文污染 |
| 邮箱式 IPC | SendMessage + teammate-message XML | 隔离通信、可追踪 |
| 权限集中 | PermissionSync 统一审批 | 安全边界不被突破 |
| 能力分类 | 只读 vs 完整权限的 Agent 类型 | 安全并行 + 受控写入 |
| 可视化渲染 | tmux/iTerm2 多 pane 着色 | 人类可观察的并行执行 |
| 共享任务列表 | blockedBy 依赖追踪 | 避免重复劳动、保证执行顺序 |

---

## 关键洞察

> **从"共享状态"到"消息传递"——每个队友的独立上下文窗口解决了上下文污染问题，邮箱式 IPC 解决了协调问题，PermissionSync 解决了安全问题。这是一套完整的分布式 Agent 操作系统雏形。**
