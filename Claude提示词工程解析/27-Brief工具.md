# 技巧二十七：Brief——不预加载的上下文注入

> 核心思想：别把所有指导都塞进系统提示词——用到的时候再注入

---

## 问题本质

系统提示词有一个根本矛盾：**写得越多，能力越强；写得越多，浪费越多。**

Claude Code 的系统提示词已经包含了身份定义、安全围栏、工具指导、行为准则……如果再把每个工具的详细状态、当前环境信息、运行模式指导全部塞进去，token 消耗会失控。

更关键的是——很多信息**只有在特定场景下才需要**。比如 Brief 工具的状态信息，只有在代理需要汇报当前状态时才用到。如果每次对话都预加载这些内容，就是在为 99% 不需要的场景买单。

**Brief 工具的解法**：不预加载，用到的时候再注入。

---

## Brief 工具：按需注入的压缩状态

Brief 工具不是一个普通的"读取状态"工具——它是一个**上下文注入管道**。当工具被调用时，它将压缩后的状态信息注入到对话上下文中。

### 导出结构

**源码**：cli.js ~line 11915

Brief 相关的导出包含三个核心常量：

```typescript
BRIEF_TOOL_NAME      // 工具名称
BRIEF_TOOL_PROMPT    // 工具被调用时注入的 prompt
BRIEF_PROACTIVE_SECTION  // 主动模式下的变体
```

这三个常量不是系统提示词的一部分——它们只在工具被调用时才进入上下文。

### 注入时机对比

```
传统模式：
  系统提示词 = 身份 + 安全 + 工具指导 + 状态信息 + ...
  → 启动时全部加载，不管用不用得到

Brief 模式：
  系统提示词 = 身份 + 安全 + 工具指导 + Brief工具注册
  → Brief 工具被调用时，才注入状态信息
```

| 维度 | 预加载模式 | Brief 按需模式 |
|------|-----------|--------------|
| Token 消耗 | 固定开销高 | 按需开销，可变 |
| 上下文占用 | 始终占用 | 仅在需要时占用 |
| 信息新鲜度 | 启动时快照，可能过时 | 调用时实时生成 |
| 灵活性 | 一种格式固定 | 可根据场景变体 |

## 惰性加载模式：不预加载，用时才注入

Brief 工具体现了一个更广泛的提示词工程模式——**惰性上下文加载**（Lazy Context Loading）。

### 核心思想

不是"万一需要就提前准备"，而是"真正需要时再加载"。

这和软件工程中的**懒加载**完全一样：

```javascript
// 传统：启动时加载所有模块
import moduleA from 'moduleA'  // 可能用不到
import moduleB from 'moduleB'  // 可能用不到
import moduleC from 'moduleC'  // 可能用不到

// 懒加载：用到时才加载
const moduleA = () => import('moduleA')  // 不用不加载
```

在提示词工程中的对应：

```typescript
// 传统：系统提示词包含所有可能需要的信息
systemPrompt = `
  ${identity}
  ${safety}
  ${toolGuide}
  ${statusInfo}        // 可能不需要
  ${envDetails}        // 可能不需要
  ${runModeGuide}      // 可能不需要
`

// 惰性注入：只注册工具，调用时才注入
systemPrompt = `
  ${identity}
  ${safety}
  ${toolGuide}
  // Brief 工具注册在这里，但内容不加载
`

// 当 Brief 被调用时：
briefContext = generateBriefStatus()  // 实时生成
injectInto(briefContext)              // 注入到当前对话
```

### 为什么这很重要

上下文窗口是有限资源。每个 token 都是成本——不仅是 API 费用成本，更是**注意力成本**。Transformer 模型对每个 token 都投入注意力，无关信息越多，关键指令的注意力越分散。

惰性加载解决的不只是 token 消耗问题，还有**注意力稀释**问题。

**类比**：就像一本百科全书——你不需要把整本都背在身上出门。真正需要查某个词条时，翻到那一页就行。Brief 工具就是那个"按需查阅"的机制。

## PROACTIVE 变体：不同场景，不同密度

Brief 工具的精妙之处在于它**不是只有一种格式**。根据代理运行模式的不同，它注入不同密度的信息。

### 前台 vs 后台

```typescript
// 前台模式：用户在看，可以交互
BRIEF_TOOL_PROMPT
  → 详细状态
  → 包含交互性指导
  → 可以输出长文本

// 后台模式（PROACTIVE）：用户不在，自主运行
BRIEF_PROACTIVE_SECTION
  → 压缩状态
  → 省略交互性指导
  → 强调关键决策点
```

这个设计的提示词工程意义：

| 场景 | 信息需求 | Brief 策略 |
|------|---------|-----------|
| 前台交互 | 用户需要详细信息做决策 | 完整格式，保留细节 |
| 后台运行 | 下次回来需要快速了解进展 | 压缩格式，只留关键 |
| 错误汇报 | 需要足够信息定位问题 | 详细格式，包含错误上下文 |
| 状态查询 | 只想知道"还在跑吗" | 极简格式，一句话状态 |

### 变体模式的工程价值

同一个工具，多种输出格式——这比"一个格式打天下"更高效。它承认了一个现实：**不同场景下，"有用"的定义完全不同。**

前台用户说"Brief 一下"，期待的是可以据此做决策的详细信息。后台代理自动 Brief 自己，需要的是"出没出问题"的二元判断。

## 更广泛的模式：工具级上下文压缩

Brief 工具不是一个孤例。它是 Claude Code 中"工具级上下文压缩"模式的一个实例。

其他类似的模式：

| 模式 | 实现 | 效果 |
|------|------|------|
| Skill 预算 | 1% 上下文窗口限制技能列表长度 | 技能列表不膨胀 |
| Agent 输出约束 | "concise report" 强制简洁 | 子 Agent 返回精简结果 |
| 工具描述裁剪 | 根据可用工具动态注入指导 | 不给不存在的工具写指导 |
| Brief 按需注入 | 调用时才加载状态信息 | 避免预加载开销 |

共同原则：**上下文是稀缺资源，只在需要时分配，分配时尽量压缩。**

---

## 提示词工程技巧总结

| 技巧 | 具体做法 | 效果 |
|------|---------|------|
| 惰性加载 | 工具调用时才注入上下文 | 减少固定 token 开销 |
| 场景变体 | 同一工具多种输出格式 | 匹配不同信息需求 |
| 注入管道 | 工具本身作为上下文注入器 | 解耦提示词和状态信息 |
| 注意力保护 | 减少无关信息 | 关键指令获得更多注意力 |
| 预算控制 | 限制各部分内容的 token 占比 | 防止任何单一部分膨胀 |

## 给抖音文案的启发

> "Claude 不会把所有知识都背在身上跟你聊天——它只带最基本的，需要什么查什么。就像你不会出门背一本字典，但手机上随时可以查。Brief 工具就是 Claude 的'按需查阅'系统——用到了才加载，不用就不浪费。"

---

## 关键洞察

上下文注入的最佳时机是"需要的时候"，不是"开始的时候"——Brief 工具用惰性加载模式，把 token 成本从固定支出变成按需支出。
