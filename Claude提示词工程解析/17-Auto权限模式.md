# 技巧十七：ML 分类器做安全门控

> 核心思想：不让主模型自己判断"我该不该执行这个"，而是派一个独立的裁判来做裁决

---

## 问题本质

AI 编程工具有一个根本矛盾：**能力越强，权限越大，风险越高**。

权限管理有三种常见策略，每种都有致命缺陷：

| 策略 | 问题 |
|------|------|
| **用户逐条确认** | 每次 `npm install` 都弹窗——效率杀手 |
| **规则白名单** | 太死板。允许 `git status` 但不允许 `git stash`？边界模糊 |
| **模型自我判断** | "你觉得这个命令安全吗？"——让执行者自己当法官，逻辑上就不自洽 |

Claude Code 的答案：**第六种方式——用一个独立的 ML 分类器来做安全判断**。

---

## 六种权限模式全景

**源码**：oa enum ~line 81750, Y5K enum

| 模式 | 行为 | 适用场景 |
|------|------|---------|
| `default` | 按内置规则判断，敏感操作弹窗 | 日常使用 |
| `acceptEdits` | 文件编辑自动放行，其他弹窗 | 批量重构 |
| `bypassPermissions` | 全部放行，不弹窗 | 受信环境，CI/CD |
| `plan` | 只读模式，禁止所有写操作 | 代码审查、架构分析 |
| `dontAsk` | 不弹窗，但也不自动放行——直接拒绝未授权操作 | 子 Agent、自动化流程 |
| `auto` | ML 分类器实时判断每条操作的安全性 | 智能权限管理 |

前五种是**确定性规则**——要么弹窗，要么放行，要么拒绝。

`auto` 是唯一引入**概率判断**的模式。

## Auto 模式：ML 分类器如何工作

### 架构

```
用户 / 模型
    │
    ▼
工具调用请求（BashTool: "rm -rf node_modules"）
    │
    ▼
┌─────────────────────────────┐
│   权限系统（oa 权限模式）     │
│                             │
│   mode = auto?              │
│     │                       │
│     ▼                       │
│   ┌─────────────────────┐   │
│   │  advisorModel        │   │ ← 独立的 ML 分类器（server-side）
│   │  ~line 12032         │   │
│   │                      │   │
│   │  输入：工具名 + 参数   │   │
│   │  输出：allow / deny   │   │
│   └─────────────────────┘   │
│                             │
│   + 延迟追踪：               │
│   turnClassifierDurationMs  │
│   turnClassifierCount       │
└─────────────────────────────┘
    │
    ▼
放行 / 拒绝 / 降级为弹窗确认
```

**源码**：advisorModel ~line 12032

### 关键设计决策

1. **独立模型**：`advisorModel` 不是主对话模型（不是 Claude 本身），而是一个专门的分类器
2. **服务端执行**：分类器跑在服务器端，不消耗用户的 token 预算
3. **实时判断**：每条工具调用都经过分类器评估
4. **延迟可观测**：`turnClassifierDurationMs` 记录每次分类耗时，`turnClassifierCount` 记录调用次数

### 为什么不用主模型自己判断？

```
❌ 让主模型判断："运行 rm -rf / 安全吗？"
   → 模型说"不安全"——但这是"建议"，不是"阻止"
   → 如果 prompt engineering 不到位，模型可能"被说服"
   → 模型同时是执行者和裁判，利益冲突

✅ 独立分类器判断："这个 Bash 命令的风险等级？
   → 分类器只看命令文本，不看对话上下文
   → 没有"被说服"的可能（没有对话能力）
   → 裁判和执行者分离，天然安全
```

## 权限决策的三种来源

**源码**：permission decision types

| 决策 | 含义 | 生效范围 |
|------|------|---------|
| `user_temporary` | 用户手动允许一次 | 仅本次调用 |
| `user_permanent` | 用户"总是允许" | 持久化到规则 |
| `user_reject` | 用户拒绝 | 仅本次调用 |

但 `auto` 模式还有一种隐式决策：**分类器决策**——用户没有参与，系统自动裁决。

### 决策优先级

```
用户手动决策（user_temporary / user_permanent / user_reject）
    ↓ 不存在时
Hooks 决策（allow / deny / ask）
    ↓ 不存在时
auto 分类器决策
    ↓ 不存在时
默认规则（default mode）
```

---

## 反馈循环：权限决策如何变成训练数据

这是最精妙的设计——**每次权限决策都在默默积累训练信号**。

```
用户使用 Claude Code
    │
    ├── 分类器自动放行 → 用户没干预 → 正样本（这操作确实安全）
    │
    ├── 分类器自动放行 → 用户手动拒绝 → 负样本（分类器判断错误）
    │
    ├── 分类器自动拒绝 → 用户手动放行 → 负样本（分类器过于保守）
    │
    └── 用户手动设置规则 → 明确信号（白名单/黑名单）
```

**这意味着**：
1. 越多人使用 `auto` 模式，分类器越精准
2. 分类器越精准，需要弹窗确认的情况越少
3. 弹窗越少，用户体验越好，更多人用 `auto`
4. 飞轮转动

### 与其他方案的对比

| 方案 | 数据来源 | 迭代速度 | 冷启动 |
|------|---------|---------|--------|
| 规则白名单 | 人工维护 | 极慢 | 需要人工初始化 |
| 模型自判断 | prompt 依赖 | 不可迭代 | 依赖 prompt quality |
| auto 分类器 | 用户行为自动收集 | 持续迭代 | 分类器预训练 |
| 强化学习 | 奖励信号 | 慢 | 需要大量交互 |

## 安全性的多层保障

`auto` 模式不是唯一的防线。Claude Code 的权限系统是**多层嵌套**的：

```
第一层：内置规则（哪些工具默认允许/拒绝）
    ↓
第二层：用户规则（settings.json 中的白名单/黑名单）
    ↓
第三层：Hooks 规则（PreToolUse 钩子的 permissionDecision）
    ↓
第四层：auto 分类器（ML 实时判断）
    ↓
第五层：兜底弹窗（以上都没决定时，问用户）
```

即使分类器误判了，还有 Hooks 可以拦截；即使 Hooks 没写，还有内置规则兜底。

**纵深防御，而非单一防线。**

---

## 延迟与成本考量

| 指标 | 含义 |
|------|------|
| `turnClassifierDurationMs` | 单次分类器调用耗时（毫秒） |
| `turnClassifierCount` | 当前轮次分类器调用次数 |

分类器是额外的网络请求，会增加每条工具调用的延迟。但从架构设计看：
- 分类器是轻量级模型（不需要理解完整对话，只判断命令安全性）
- 服务端执行，不增加客户端 token 消耗
- 可以缓存常见命令的判断结果

---

## 关键洞察

> **安全判断不应该由执行者自己做——这是审计学的基本原则，也是 AI 安全的基本原则。**

`auto` 模式的核心价值不是"更方便"，而是"更安全"：它引入了一个**独立于主模型的裁判**，并通过用户行为反馈持续进化。模型只负责"怎么做"，分类器负责"该不该做"——职责分离，权限才真正可控。
