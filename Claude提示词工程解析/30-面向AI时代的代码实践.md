# 技巧三十：面向 AI 时代的代码实践——从 30 篇分析中提炼的 12 条原则

> 核心思想：让 AI 读得懂、搜得到、改得动的代码，本身就是更好的代码

---

## 为什么这个话题重要

30 篇分析拆解了 Claude Code 的提示词工程、Hooks 系统、权限管线、沙箱架构……但有一个反直觉的发现贯穿始终：

**Claude Code 对代码的要求，恰好是人类一直应该做但没做到的事。**

文件不要太长、命名要清晰、依赖要低耦合、日志要有锚点——这些都是教科书里的原则。AI 只是让违反这些原则的代价变得可量化：一个 5000 行的文件，人类可以硬读，AI 需要 3 次 Read 操作，消耗 5 万字符，还可能在上下文压缩后丢失。

这不是"为 AI 写代码"——是"用 AI 的硬约束，倒逼我们写出本来就应该写的代码"。

---

## 第一层：结构约束——AI 的"阅读能力"有硬上限

### 原则一：文件 200-800 行，绝不超过 2000 行

AI 工具有三个硬限制（来自第 12 篇）：
- 单次 Read 最多 2000 行
- 单文件最大 256 KB
- 单次读取最多 25,000 tokens

一个 5000 行的文件需要 3+ 次 Read，消耗约 5 万字符——占单条消息预算的一半。Claude Code 自己最大的工具（BashTool）跨 18 个文件共 12,429 行，而不是一个单体文件。

| 文件大小 | AI 读取体验 | 人类维护体验 |
|---------|-----------|-----------|
| 200-800 行 | 一次 Read 完整加载 | 单屏可理解全貌 |
| 800-2000 行 | 需要 context 参数 | 需要滚动才能理解 |
| 2000+ 行 | 多次 Read + 压缩后可能丢失 | 已经难以维护 |

**行动规则**：按职责拆分，不是按行数拆分。Claude Code 的每个工具都有独立目录（`src/tools/BashTool/`），逻辑、UI、提示词、权限各自成文。

### 原则二：一个功能 = 一个目录 + 一个 index.ts

Claude Code 的工具目录结构是典范：
```
src/tools/BashTool/
  ├── BashTool.ts      # 工具定义 + 执行逻辑
  ├── UI.tsx           # 终端渲染
  ├── prompt.ts        # 提示词注入
  ├── permissions.ts   # 权限模型
```

`Glob("src/tools/BashTool/**")` 一次调用返回该工具的所有文件。而扁平目录下 36 个工具挤在一个文件夹里，AI 必须逐个文件名猜测关联。

**第 15 篇的核心公式**：高内聚让 Glob 精确，低耦合减少搜索链深度。一条 5 跳的依赖链（A→B→C→D→E）需要 5+ 次搜索；清晰接口边界的低耦合设计只需要 2 次。

**index.ts 是代码地图**——AI 先读它（通常 200 行以内），再导航到具体实现文件。没有它，AI 必须逐个检查 50 个文件才能发现公共 API。

## 第二层：命名即索引——名字的双重职责

### 原则三：命名要足够独特，一次 Grep 就能找到

第 14 篇和第 15 篇的对比：

| 搜索词 | 结果质量 |
|--------|---------|
| `buildTool\(` | 精确找到所有工具定义 |
| `satisfies Command` | 精确找到所有命令定义 |
| `process()` | 溢出——多个模块都有这个名字 |
| `handle()` | 溢出——泛化名称，瞬间命中 250 行结果 |

**行动规则**：
- 避免 `process()`、`handle()`、`run()` 等泛化名称
- 优先使用 `authenticateUser()`、`buildTool()`、`compileSystemPrompt()` 等有辨识度的名称
- 严格遵循命名约定：文件 kebab-case，类 PascalCase，函数 camelCase，常量 UPPER_SNAKE_CASE

### 原则四：日志里嵌入函数名，创造搜索锚点

```
好：logger.error('authenticateUser: token validation failed', { userId })
差：logger.error('auth failed', { userId })
```

好的版本支持渐进式搜索：从 `Grep("authenticate")`（300 条，截断）到 `Grep("authenticateUser: token validation failed")`（1-3 条，精确）。差的版本无法进一步缩小范围。

## 第三层：反过度工程——少即是多

### 原则五：三行相似的代码 > 过早的抽象

第 3 篇直接引用了 Claude Code 系统提示词的原文：
- "不要为不会发生的场景添加错误处理、回退或验证"
- "能直接改代码时，不要加特性开关或向后兼容垫片"
- "三行相似的代码好过一个过早的抽象"
- "不要为一次性操作创建辅助函数、工具或抽象"

这背后的逻辑：过早的抽象迫使 AI（和人）穿越间接层才能理解实际发生了什么。而重复代码是局部可读的——每一份拷贝都可以独立理解，不需要交叉引用。

**抽象的正确时机**：同一模式出现 3+ 次，且抽象让每个使用点更容易理解时。Claude Code 的 `SHARED_PREFIX` 模式就是正确示例——相同的指令出现在 6+ 个 Agent 提示词中，提取为常量后改一处即改全部。

### 原则六：只在系统边界做验证

用户输入、外部 API 响应——这些是需要验证的地方。内部函数之间的调用不需要。类型系统（TypeScript）已经帮你做了大部分工作。

如果你发现自己在写"这段代码理论上不可能到达"的防御性分支——删掉它。它对人类是噪音，对 AI 是干扰。

## 第四层：文档策略——地图而非百科全书

### 原则七：CLAUDE.md 是目录，不是仓库

第 15 篇给出了具体约束：
- 单文件最大 40,000 字符
- `@include` 递归最多 5 层
- MEMORY.md 条目：200 行 / 25KB 上限

最佳实践是列出关键文件和导航路径，不是贴代码：

```markdown
## 关键文件
| 文件 | 职责 |
|------|------|
| src/QueryEngine.ts | LLM 引擎——流式输出、工具调用循环 |
| src/Tool.ts | 工具类型/接口、buildTool() 工厂 |
```

**200 行以内**。超过这个长度，记忆系统本身就在消耗过多上下文。

### 原则八：验证步骤 > 使用说明

文档不应只说"运行测试"，而应说"运行 `npm test` 并确认全部 47 个测试通过。如有失败，查看 `test-results.log`"。

第 10 篇揭示了一个反直觉的事实：**更强的模型更擅长撒谎**。Capybara v8 的虚假声明率是 29-30%（v4 只有 16.7%），因为它能编造更可信的解释。

应对策略来自第 5 篇的验证代理设计：`disallowedTools: ['Agent', 'PlanMode', 'Edit', 'Write']`——它在物理上无法修改代码，只能验证。这种"读写分离"同样适用于人类团队。

## 第五层：事件驱动 > 文档驱动

### 原则九：重复的规则变成钩子，而非提示词

第 16 篇是整个系列最深刻的架构洞察。对比：

**文档驱动（消耗 token，不可靠）**：
```
你是一个细心的开发者。每次编辑文件后，你必须运行 linter。
不要忘记。这是强制性的。运行：npx eslint <file>
```
每轮对话 ~50 tokens，模型可能"忘记"，需要反复强调。

**事件驱动（零 token，100% 可靠）**：
```json
{
  "PostToolUse": {
    "matcher": "Edit",
    "hooks": [{ "type": "command", "command": "npx eslint $file", "async": true }]
  }
}
```

**行动规则**：如果你发现自己在文档或注释中反复强调同一条规则——问自己：这条规则能不能变成一个 CI 检查、linter 规则、类型约束？如果能，就把它从文档中移出去。

### 原则十：能用类型系统约束的，不要用注释描述

第 19 篇的沙箱架构和第 18 篇的 MCP 注解语义层揭示了同一个模式：

```
自然语言："此端点不修改数据" → 模型猜测："可能安全" → 不同模型猜不同结果
结构化注解：{ readOnlyHint: true } → 权限系统确定性判断 → 不需要模型猜测
```

TypeScript 的类型系统、linter 规则、测试断言——这些都是"结构化注解"。它们比任何注释都可靠，因为它们是**可执行的**。

## 第六层：AI 协作模式——为"从零探索"而设计

### 原则十一：代码应该不依赖 IDE 就能被理解

第 12 篇揭示了一个关键事实：Claude Code **不做任何预处理**。没有 RAG、没有向量搜索、没有代码索引、没有 Tree-sitter 解析（Bash 除外）。AI 每次都从零开始，用"搜索→阅读→判断"模式工作。

这意味着：如果你的代码只能在 IDE 的 go-to-definition 和 find-references 帮助下才能理解——AI 看到的就是一坨乱麻。

**行动规则**：
- `Grep("functionName")` 和 `Glob("src/feature/**")` 应该足以导航整个代码库
- 每个功能模块应该独立可理解，不需要加载整个代码库
- 子代理的输出应该是结构化的（命令/输出/通过|失败），方便上层解析

### 原则十二：读可以并行，写必须串行

第 5 篇的多代理架构和第 22 篇的邮箱式 IPC 揭示了一个核心模式：

1. 只读代理（Explore）可以安全并行——没有副作用
2. 写入代理（General-Purpose）必须串行——避免冲突
3. 每个代理有独立的上下文窗口——避免上下文污染
4. 代理之间通过结构化消息通信，不共享状态

这同样适用于人类团队：多个开发者可以同时阅读代码，但对同一文件的修改需要协调。

## 第七层：六层防御模型——终极洞察

整个系列最深刻的认识来自选题文档：

```
第 1 层：提示词层      ——用语言引导行为
第 2 层：Hooks 外置层   ——用事件替代 token 消耗
第 3 层：权限/分类器层   ——用 ML 和协议语义替代模型猜测
第 4 层：沙箱物理层     ——用 OS 级隔离替代提示词约束
第 5 层：实验平台层     ——用 A/B 测试替代经验调优
第 6 层：可观测性层     ——用数据反馈替代主观评估
```

**核心原则：能在更低层约束的，就不要在更高层描述。**

- OS 能阻止写 `.git/`，提示词就不需要提
- 类型系统能防 null 错误，文档就不需要警告
- Linter 能强制格式，风格指南就不需要描述
- CI 能检查测试通过，PR 描述就不需要写"已测试"

## 关键洞察

> **AI 时代的代码实践不是新发明——它是旧原则的量化执行。AI 的硬约束（2000 行上限、25K token 读取限制、无索引搜索）把"应该做"变成了"必须做"。写 AI 友好的代码，本质上就是写本来就应该写的代码。**
