# 技巧六：输出风格覆盖层

> 核心思想：同一引擎，通过"风格覆盖"切换不同输出模式

---

## 架构

`src/constants/outputStyles.ts` 定义了一套**输出风格覆盖系统**：

```
┌────────────────────────────┐
│     基础系统 Prompt          │
│  (身份 + 行为 + 工具规则)    │
└────────────┬───────────────┘
             │
    可选覆盖 ▼
┌────────────────────────────┐
│  Explanatory Mode          │  教育模式：加知识块
│  Learning Mode             │  学习模式：让用户动手
│  Custom (plugin/user)      │  自定义
└────────────────────────────┘
```

### 优先级链

```
plugin forced > managed > project > user > built-in
```

- **plugin forced**: 插件强制设置的风格
- **managed**: 组织管理的风格
- **project**: 项目级 CLAUDE.md 中定义的风格
- **user**: 用户个人偏好
- **built-in**: 内置默认风格

---

## 两种内置覆盖

### Explanatory Mode（解释模式）

在每次代码写入/编辑后，自动插入知识块：

```
★ Insight ─────────────────────────────
  Act 1: What does this code do?
  Act 2: Why this specific approach?
  Act 3: What's the broader implication?
────────────────────────────────────────
```

**设计细节**：
- 使用 `★` 符号和分隔线制造视觉标记
- 每个 insight 块包含 2-3 个教育性观点
- 只在代码写入/编辑时触发，不影响其他交互
- 观点要求："surprising or non-obvious"（不是复述代码做了什么）

**触发条件**：只在 `Write`, `Edit`, `NotebookEdit` 工具调用后触发。

### Learning Mode（学习模式）

完全改变交互模式——AI 不直接写代码，而是引导用户写：

```
Context: [解释背景和目标]
Your Task: [具体要写什么]
Guidance: [提示和方向]

TODO(human): [等待用户在这里写代码]
```

**核心机制**：
- `TODO(human)` 标记：AI 插入这个标记后**暂停**，等用户写代码
- "Learn by Doing" 格式：Context → Your Task → Guidance → 用户动手
- AI 会在用户写完后 review 并提供反馈

**限制**：
- "No emojis unless the user asks for it" —— 学习模式也遵守基础风格规则
- "Verify user code before marking tasks complete" —— AI 不能跳过验证

---

## 自定义 Output Style

用户/插件可以注册自定义风格：

```typescript
interface OutputStyle {
  name: string
  content: string  // 追加到系统 prompt 的内容
  // ...
}
```

自定义风格的内容会被追加到系统 prompt 的末尾，不影响基础行为准则。

---

## 提示词工程技巧总结

| 技巧 | 具体做法 | 效果 |
|------|---------|------|
| 覆盖层模式 | 追加内容而非替换基础 prompt | 保持核心行为不变，只改变输出形态 |
| 触发条件绑定 | insight 块只在写代码后触发 | 不是每条回复都加，精准控制 |
| TODO 标记 | `TODO(human)` 作为暂停信号 | AI 主动让位给用户 |
| 优先级链 | plugin > managed > project > user > built-in | 多层覆盖，最后设置的赢 |
| 视觉标记 | `★ Insight ──` 分隔线 | 在长回复中易于识别 |
