# 技巧二十八：受控扩展——插件如何注入提示词

> 核心思想：核心提示词是宪法，插件是法律——在宪法框架内扩展，而不是无限膨胀

---

## 问题本质

Claude Code 的系统提示词已经是一份工程杰作——分层组装、特性门控、条件注入，所有部分都精心设计。但有一个根本问题：**核心提示词不能无限膨胀，但用户需求千差万别。**

- 用户 A 需要 Claude 支持 Rust 代码审查
- 用户 B 需要 Claude 集成 Jira 任务管理
- 用户 C 需要 Claude 懂公司内部的代码规范
- 企业 D 需要禁止所有第三方扩展

如果把这些需求都塞进核心提示词，系统提示词会变成一本百科全书——每次对话消耗几万 token 只加载可能用不到的指导。

**插件系统**（Plugin System）解决这个矛盾的方式是：**核心保持精简，扩展通过插件动态注入。**

---

## 插件架构：Marketplace → Plugin → Skills/Tools/Agents

### 三层结构

```
┌─────────────────────────────────────────┐
│           Marketplace（市场）             │
│   插件的来源，类似 npm registry          │
├─────────────────────────────────────────┤
│           Plugin（插件）                  │
│   一个功能包，包含技能、工具、代理        │
├─────────────────────────────────────────┤
│    Skills / MCP Servers / Agents         │
│    插件注入的具体能力                     │
└─────────────────────────────────────────┘
```

### 插件可以注入什么

插件不只是"加一个命令"——它可以向系统提示词注入**四种能力**：

| 注入类型 | 效果 | 类比 |
|---------|------|------|
| 自定义 Skills | 添加 `/command` 斜杠命令 | 给编辑器装插件命令 |
| MCP Servers | 添加新的工具和数据源 | 给系统装新硬件驱动 |
| 自定义 Agents | 添加新的子代理类型 | 给公司招新岗位 |
| Output Style Overlays | 覆盖输出风格 | 换一套 UI 主题 |

### 启用方式

插件通过 `settings.json` 配置启用：

```json
{
  "enabledPlugins": [
    "plugin-id@marketplace-id",
    "another-plugin@another-marketplace"
  ]
}
```

**源码**：cli.js ~line 691, ~line 12032

格式 `plugin-id@marketplace-id` 的设计意图：每个插件必须声明来源市场。这不仅是命名空间隔离，更是**安全追溯**——出了问题可以定位到是哪个市场提供的插件。

## 企业管控：白名单/黑名单机制

插件系统的精妙之处在于它不只是一个"开放生态"——它同时支持**企业级管控**。

### 两层管控

```typescript
// 白名单：只允许来自这些市场的插件
strictKnownMarketplaces = [
  'anthropic-official',
  'verified-partner-1',
  // ...
]

// 黑名单：明确禁止这些市场
blockedMarketplaces = [
  'untrusted-marketplace-x',
  // ...
]
```

### 管控矩阵

| 企业场景 | 白名单配置 | 黑名单配置 | 效果 |
|---------|-----------|-----------|------|
| 严格管控 | 只允许官方市场 | — | 只能用 Anthropic 官方插件 |
| 宽松管控 | 允许认证市场 | 屏蔽已知风险市场 | 大部分插件可用 |
| 开放模式 | 不设白名单 | 只屏蔽已知恶意 | 最大灵活性 |
| 零信任 | 不允许任何市场 | — | 禁用整个插件系统 |

**类比**：就像手机的应用商店策略——个人用户想装什么装什么（开放模式），但企业配发的手机只能装公司批准的应用（严格管控）。同一个插件系统，不同的安全策略。

### 插件级配置

管控甚至可以细化到单个插件级别：

```json
{
  "pluginConfigs": {
    "some-plugin@some-marketplace": {
      "mcpServerSettings": {
        "timeout": 30000,
        "allowedTools": ["read", "search"]
      }
    }
  }
}
```

这意味着企业可以：允许某个插件安装，但限制它能使用的工具。**"你可以装，但你不能什么都做。"**

## 动态注入：插件内容如何进入系统提示词

插件安装后，它的内容不是静态合并到系统提示词中的——而是**动态注入**。

### 注入管道

```
启动时：
  1. 读取 settings.json 中的 enabledPlugins
  2. 验证每个插件的 marketplace 是否在白名单内
  3. 加载插件声明的 Skills、MCP Servers、Agents
  4. 将这些声明注入到系统提示词的相应位置

运行时：
  - Skills 出现在技能列表中，可用 /command 调用
  - MCP Servers 的工具出现在工具列表中
  - Agents 可作为子代理被派生
```

### 注入位置

插件内容不是随意拼接的——它们被注入到系统提示词的**特定结构化位置**：

```typescript
// 系统提示词组装
systemPrompt = [
  identitySection,        // 身份定义（静态）
  safetySection,          // 安全围栏（静态）
  toolSection,            // 工具指导（动态：根据可用工具）
  skillSection,           // 技能列表（动态：包含插件技能）
  agentSection,           // 代理定义（动态：包含插件代理）
  outputStyleSection,     // 输出风格（可被插件覆盖）
].join('\n')
```

插件的 Skills 和 Agents 被追加到对应的动态 section 中，和内置能力**同等对待**。模型看到的是一个统一的能力列表——它不需要区分"这个技能是内置的还是插件装的"。

**源码**：cli.js ~line 691（inlinePlugins）

### 斜杠命令的自动注册

插件的 Skills 自动以 `/command` 形式出现在用户界面：

```
内置：
  /help, /clear, /config, ...

插件：
  /review-rust    ← Rust 审查插件
  /jira-sync      ← Jira 同步插件
  /company-lint   ← 公司规范检查插件
```

用户不需要知道哪些是内置的，哪些是插件的——体验上完全一致。

## Cowork 插件：协作模式扩展

插件系统还支持一种特殊类型——**Cowork 插件**，用于协作操作：

```
Cowork 插件：
  - 可以注册为协作代理（与主 Agent 并行工作）
  - 可以定义自己的通信协议
  - 可以参与多代理任务分配
```

这把插件系统从"加功能"升级到了"加角色"——插件不只是工具箱里的新工具，还可以是团队里的新成员。

## 提示词工程技巧总结

| 技巧 | 具体做法 | 效果 |
|------|---------|------|
| 分层架构 | Marketplace → Plugin → 能力 | 清晰的依赖和来源管理 |
| 白名单/黑名单 | strictKnownMarketplaces / blockedMarketplaces | 企业级安全管控 |
| 命名空间隔离 | plugin-id@marketplace-id | 安全追溯和冲突避免 |
| 动态注入 | 插件内容追加到对应 section | 核心提示词保持精简 |
| 统一接口 | 插件技能与内置技能同等对待 | 用户体验一致 |
| 细粒度管控 | pluginConfigs 限制单个插件权限 | 允许安装但限制能力 |

## 给抖音文案的启发

> "Claude 的提示词就像一部宪法——规定了基本权利和底线。但具体怎么做生意、怎么审查代码、怎么管理项目，这些不是宪法该管的事。插件系统就是'法律'——在宪法框架内，给 Claude 装上你需要的专业能力。而且老板可以决定：哪些'法律'可以装，哪些绝对不行。"

---

## 关键洞察

核心提示词是宪法，插件是法律——受控扩展而非无限膨胀，同一个系统在个人用户手中是开放生态，在企业手中是受控平台。
